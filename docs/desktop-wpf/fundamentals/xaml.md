# <a name="xaml-overview-in-wpf"></a>Обзор XAML в WPF

В этой статье описаны особенности языка XAML и показано, как можно использовать XAML для написания приложений Windows Presentation Foundation (WPF). В этой статье конкретно описывается XAML как реализованная WPF. XAML сама по себе является более широкой языковой концепцией, чем WPF.

## <a name="what-is-xaml"></a>Что такое XAML

XAML — это декларативный язык разметки. В соответствии с моделью программирования .NET Core XAML упрощает создание интерфейса для приложения .NET Core. Можно создать видимые элементы uI в декларативной разметке XAML, а затем отделить определение uI от логики времени выполнения, используя файлы с задинкой кода, которые соединяются с разметкой через частичные определения класса. Язык XAML напрямую представляет создание экземпляров объектов в конкретном наборе резервных типов, определенных в сборках. В этом заключается его отличие от большинства других языков разметки, которые, как правило, представляют собой интерпретируемые языки без прямой связи с системой резервных типов. XAML позволяет работать, где отдельные стороны могут работать над uI и логикой приложения, используя потенциально разные инструменты.

При представлении в виде текста файлы XAML являются XML-файлами, которые обычно имеют расширение `.xaml`. Файлы можно сохранять в любой кодировке, поддерживаемой XML, но обычно используется кодировка UTF-8.

### <a name="collection-syntax"></a>Синтаксис коллекций

Язык XAML включает некоторые оптимизации, делающие разметку более удобной для чтения. Одна из оптимизаций заключается в том, что, если определенное свойство принимает тип коллекции, элементы, объявленные в разметке как дочерние в пределах значения этого свойства, становятся частью коллекции. В этом случае коллекция дочерних объектных элементов является значением, которое задается для свойства коллекции.

В следующем примере показан синтаксис <xref:System.Windows.Media.GradientBrush.GradientStops%2A> сбора для установления значений свойства.

```xaml
<LinearGradientBrush>
  <LinearGradientBrush.GradientStops>
    <!-- no explicit new GradientStopCollection, parser knows how to find or create -->
    <GradientStop Offset="0.0" Color="Red" />
    <GradientStop Offset="1.0" Color="Blue" />
  </LinearGradientBrush.GradientStops>
</LinearGradientBrush>
```

### <a name="xaml-content-properties"></a>Свойства содержимого в языке XAML

XAML определяет языковую функцию, в соответствии с которой класс может назначить именно одно из своих свойств свойством _содержимого_ XAML. Дочерние элементы данного объектного элемента используются для задания значения этого свойства содержимого. Другими словами, для свойства содержимого (и только для него) можно опустить элемент свойства, указав это свойство в XAML-разметке, и тем самым создать более наглядную метафору родительского/дочернего элементов в разметке.

Например, <xref:System.Windows.Controls.Border> указывается _свойство содержимого_ <xref:System.Windows.Controls.Decorator.Child%2A>. Следующие <xref:System.Windows.Controls.Border> два элемента рассматриваются одинаково. Первый элемент использует преимущества синтаксиса свойства содержимого и опускает элемент свойства `Border.Child`. Второй пример явно показывает `Border.Child`.

```xaml
<Border>
  <TextBox Width="300"/>
</Border>
<!--explicit equivalent-->
<Border>
  <Border.Child>
    <TextBox Width="300"/>
  </Border.Child>
</Border>
```

Для языка XAML действует следующее правило: значение свойства содержимого XAML должно быть задано полностью либо до, либо после любых других элементов свойств данного объектного элемента. Например, следующая разметка не компилируется.

```xaml
<Button>I am a
  <Button.Background>Blue</Button.Background>
  blue button</Button>
```

Для получения дополнительной информации о специфике синтаксиса XAML [см.](../../framework/wpf/advanced/xaml-syntax-in-detail.md)

## <a name="case-and-white-space-in-xaml"></a>Корпус и белое пространство в XAML

В целом, XAML является чувствительным к случаям. Для решения типов поддержки WPF XAML является чувствительным к случаям по тем же правилам, что CLR является чувствительным к случаям. Объектные элементы, элементы свойств и имена атрибутов должны быть указаны с учетом регистра символов при сравнении по имени с базовым типом в сборке или с элементом типа. Ключевые слова и примитивы языка XAML также чувствительны к случаям. Значения не всегда чувствительны к случаям. Учет регистра для значений будет зависеть от поведения преобразователя типов, связанного со свойством, которое принимает значение, или с соответствующим типом значения свойства. Например, свойства, <xref:System.Boolean> которые принимают `true` тип `True` может принимать либо или как эквивалентные значения, но только потому, что родной WPF XAML парсер тип преобразования для строки <xref:System.Boolean> уже позволяет их в качестве эквивалентов.

Процессоры и сериализаторы WPF XAML будут игнорировать или упускать все незначительное белое пространство и нормализуют любое значительное белое пространство. Это согласуется с рекомендациями поведения белого пространства по умолчанию спецификации XAML. Такое поведение имеет последствие только тогда, когда вы указываете строки в свойствах содержимого XAML. Проще говоря, XAML преобразует символы пространства, линейного питания и вкладок в пробелы, а затем сохраняет одно пространство, если находится на обоих концах смежной строки. Полное объяснение обработки белого пространства XAML не рассматривается в этой статье. Для получения дополнительной информации [см.](../xaml-services/white-space-processing.md)

## <a name="markup-extensions"></a>Расширения разметки

Расширения разметки являются понятием языка XAML. При использовании для предоставления значения синтаксиса атрибутов фигурные скобки (`{` и `}`) указывают на использование расширения разметки. При обработке XAML расширения разметки обрабатываются иначе, чем значения атрибутов, обычно рассматриваемые как литеральная строка или как значение, которое может быть преобразовано в строку.

Наиболее распространенными расширениями разметки, используемыми в программировании приложений WPF, являются, [`Binding`](../../framework/wpf/advanced/binding-markup-extension.md)используемые для выражения связывания данных, а также ссылки на ресурсы [`StaticResource`](../../framework/wpf/advanced/staticresource-markup-extension.md) и [`DynamicResource`](../../framework/wpf/advanced/dynamicresource-markup-extension.md). Расширения разметки часто используют промежуточные типы выражений для включения таких функций, как отсрочка значений или ссылки на другие объекты, которые присутствуют только во время выполнения.

## <a name="type-converters"></a>Преобразователи типа

Корневой элемент также содержит атрибуты `xmlns` и `xmlns:x`. Эти атрибуты указывают обработчику XAML, какие пространства имен XAML содержат определения типов для резервных типов, на которые будет ссылаться разметка в качестве элементов. Атрибут `xmlns` специально указывает пространство имен XAML по умолчанию. В пределах пространства имен XAML по умолчанию объектные элементы в разметке можно указывать без префикса. Для большинства сценариев приложений WPF, а также для почти всех примеров, приведенных в разделах WPF SDK, пространство имен XAML по умолчанию отображается в пространстве `http://schemas.microsoft.com/winfx/2006/xaml/presentation`имен WPF. Атрибут `xmlns:x` указывает дополнительное пространство имен XAML, которое сопоставлено пространству имен языка `http://schemas.microsoft.com/winfx/2006/xaml`.

Это использование атрибутов `xmlns` для определения области использования и сопоставления области видимости имен согласовано со спецификацией XML 1.0. Области видимости имен XAML отличаются от областей видимости имен XML только тем, что область видимости имен XAML также подразумевает некоторую информацию о резервном копировании элементов области видимости имен по типам в случае разрешения типов и анализа XAML.

Атрибуты `xmlns` строго необходимы только на корневом элементе каждого файла XAML. Определения `xmlns` будут применяться ко всем элементам-потомкам корневого элемента (такое поведение также соответствует спецификации XML 1.0 для `xmlns`.). Атрибуты `xmlns` также разрешены для других элементов ниже корня и будут применены ко всем элементам-потомкам определяющего элемента. Впрочем, частое определение или повторное определение пространств имен XAML может сделать стиль разметки XAML трудным для человеческого восприятия.

Внедрение процессора XAML WPF включает в себя инфраструктуру, которая будет осведомлена о основных сборках WPF. Известно, что основные сборки WPF содержат типы, поддерживающие отображение WPF в пространстве имен по умолчанию XAML. Это осуществляется за счет конфигурации, являющейся частью файла построения проекта, построения WPF и систем проекта. Таким образом, объявление пространства имен XAML по умолчанию `xmlns` является всем необходимым для того, чтобы ссылаться на элементы XAML, которые приходят из сборок WPF.

### <a name="the-x-prefix"></a>X: префикс

В предыдущем примере корневого элемента префикс `x:` использовался для сопоставления пространства имен XAML `http://schemas.microsoft.com/winfx/2006/xaml`, которое является выделенным пространством имен XAML, поддерживающим конструкции языка XAML. Эта `x:` приставка используется для отображения этого пространства имен XAML в шаблонах для проектов, в примерах и в документации по всему SDK. Пространство имен XAML для языка XAML содержит несколько конструкций программирования, которые вы будете часто использовать в XAML. Ниже приведен список наиболее распространенных конструкций программирования с префиксом `x:`, которые могут вам потребоваться.

- [x:Key](../xaml-services/xkey-directive.md): Устанавливает уникальный ключ <xref:System.Windows.ResourceDictionary> для каждого ресурса в (или аналогичных словарных концепциях в других средах). `x:Key`вероятно, составит 90 процентов `x:` обычаев, которые вы увидите в разметке типичного приложения WPF.

- [x:Класс](../xaml-services/xclass-directive.md): Определяет пространство имен CLR и название класса для класса, который обеспечивает код-за для страницы XAML. Согласно модели программирования WPF, такой класс должен присутствовать для поддержки кода программной части, поэтому сопоставление `x:` присутствует почти всегда, даже если ресурсы отсутствуют.

- [x:Name](../xaml-services/xname-directive.md). Задает имя объекта во время выполнения для экземпляра, который существует в коде во время выполнения после обработки объектного элемента. Как правило, на практике часто используется свойство, определенное в WPF, эквивалентное [x:Name](../xaml-services/xname-directive.md). Такие свойства отображаются специально к свойству поддержки CLR и, таким образом, более удобны для программирования приложений, где вы часто используете код времени выполнения для поиска названных элементов из инициализированного XAML. Наиболее распространенным <xref:System.Windows.FrameworkElement.Name%2A?displayProperty=nameWithType>таким свойством является . Вы все еще можете использовать [x:Name,](../xaml-services/xname-directive.md) <xref:System.Windows.FrameworkElement.Name%2A> когда эквивалентное свойство уровня WPF не поддерживается в определенном типе. Это происходит в некоторых сценариях анимации.

- [x:Static](../xaml-services/xstatic-markup-extension.md). Позволяет использовать ссылку, возвращающую статическое значение, в остальном не являющееся свойством, совместимым с XAML.

- [x:Тип](../xaml-services/xtype-markup-extension.md): Строит <xref:System.Type> ссылку на основе имени типа. Это используется для указания атрибутов, <xref:System.Type> <xref:System.Windows.Style.TargetType%2A?displayProperty=nameWithType>которые принимают, хотя часто свойство<xref:System.Type> имеет родной строки к-конверсии таким образом, что [x:Type](../xaml-services/xtype-markup-extension.md) разметки расширение использования является необязательным.

Существуют дополнительные конструкции программирования в пространстве имен XAML/с префиксом `x:`, которые применяются не так часто. Дополнительные сведения см. в разделе [Возможности пространства имен языка XAML (x:)](../xaml-services/namespace-language-features.md).

## <a name="custom-prefixes-and-custom-types-in-xaml"></a>Пользовательские префиксы и пользовательские типы в XAML

Для ваших пользовательских сборок или для сборок за пределами ядра WPF **ВИСКор,** **Презентация Испиративная** система и **WindowsBase**— вы можете указать сборку как часть пользовательского `xmlns` отображения. Затем можно указывать в пользовательском XAML-коде ссылки на типы из этой сборки, если в типе правильно реализована поддержка требуемых средств XAML.

Ниже приводится основной пример того, как пользовательские префиксы работают в разметке XAML. Префикс `custom` определяется в теге корневого элемента и отображается в определенной сборке, которая упакована и доступна с приложением. Эта сборка содержит тип `NumericUpDown`, который реализуется для поддержки общего использования XAML, а также использования наследования класса, которое можно вставить в данную конкретную точку модели содержимого XAML WPF. Экземпляр этого элемента управления `NumericUpDown` объявлен как объектный элемент, префикс обеспечивает, что анализатор XAML "знает", какое пространство имен XAML содержит тип и, следовательно, где находится резервная сборка, содержащая определение типа.

```xaml
<Page
    xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
    xmlns:custom="clr-namespace:NumericUpDownCustomControl;assembly=CustomLibrary"
    >
  <StackPanel Name="LayoutRoot">
    <custom:NumericUpDown Name="numericCtrl1" Width="100" Height="60"/>
...
  </StackPanel>
</Page>
```

Дополнительные сведения о пользовательских типах в языке XAML см. в разделе [Код XAML и пользовательские классы для WPF](../../framework/wpf/advanced/xaml-and-custom-classes-for-wpf.md).

## <a name="xaml-named-elements"></a>Названные элементы XAML

По умолчанию экземпляр объекта, созданный в графе объекта путем обработки объектного элемента XAML, не имеет уникального идентификатора или ссылки на объект. Напротив, если конструктор вызывается в коде, результат конструктора почти всегда используется для указания созданного экземпляра в качестве значения переменной, чтобы на него можно было ссылаться в дальнейшем коде. Для предоставления стандартизованного доступа к объектам, созданным с помощью определения разметки, в языке XAML определяется [атрибут x:Name](../xaml-services/xname-directive.md). Можно задать значение атрибута `x:Name` для любого объектного элемента. В коде программной части выбранный идентификатор эквивалентен переменной экземпляра, которая ссылается на созданный экземпляр. Во всех отношениях названные элементы функционируют так, как если бы они были объектами экземпляров (ссылки на имя этого экземпляра), а за кодом можно ссылаться на названные элементы для обработки взаимодействий времени выполнения в приложении. Эта связь между экземплярами и переменными достигается компилятором разметки WPF XAML и более конкретно включает в себя функции и шаблоны, такие как, <xref:System.Windows.Markup.IComponentConnector.InitializeComponent%2A> которые не будут подробно обсуждаться в этой статье.

Элементы XAML уровня WPF <xref:System.Windows.FrameworkElement.Name%2A> наследуют свойство, что эквивалентно определенному `x:Name` атрибуту XAML. Некоторые другие классы также предоставляют эквиваленты уровня свойств для `x:Name`, который также обычно определяется как свойство `Name`. Значения `x:Name` будут предоставлять идентификатор элемента XAML, который может быть использован во время выполнения, <xref:System.Windows.FrameworkElement.FindName%2A>либо конкретными подсистемами, либо утилитами, такими как.

Как и в случае с переменными, имена экземпляров классов XAML регулируются понятием области, так что обеспечивается уникальность имен в пределах определенной области с известными границами. Основная разметка, определяющая страницу, обозначает одну уникальную область видимости имен XAML с корневым элементом страницы в качестве границы. Однако другие источники разметки могут взаимодействовать со страницей во время выполнения, такие как стили или шаблоны в стилях, и такие источники разметки часто имеют свои собственные xAML namescopes, которые не обязательно соединяются с XAML namescope страницы. Для получения `x:Name` дополнительной информации о и <xref:System.Windows.FrameworkElement.Name%2A>XAML namescopes, см., [x:Name Directive](../xaml-services/xname-directive.md), или [WPF XAML Namescopes](../../framework/wpf/advanced/wpf-xaml-namescopes.md).

## <a name="attached-properties-and-attached-events"></a>Прилагаемые свойства и прилагаемые события

Одна из возможностей языка XAML позволяет задавать для любого элемента определенные свойства или события, независимо от того, существует ли свойство или событие в таблице определений типа для этого элемента. Версия свойств этой возможности называется вложенным свойством, версия событий называется вложенным событием. Концептуально можно представить вложенные свойства и события как глобальные члены, которые могут быть заданы для любого элемента/экземпляра объекта XAML. Впрочем, этот элемент/класс (или более крупная инфраструктура) должен поддерживать резервное хранилище свойств для вложенных значений.

Вложенные свойства в языке XAML обычно используются через синтаксис атрибутов. В синтаксисе атрибутов указывается прилагаемое свойство в форме. `ownerType.propertyName`

На первый взгляд, это напоминает использование элемента `ownerType` свойства, но в этом случае вы указываете всегда другой тип, чем элемент объекта, где устанавливается прилагаемое свойство. `ownerType`— это тип, который предоставляет методы доступа, необходимые процессору XAML для получения или установки прилагаемых значений свойства.

Наиболее распространенным сценарием для вложенных свойств является разрешение дочерним элементам предоставления значения свойства их родительскому элементу.

Класс <xref:System.Windows.Controls.DockPanel> также включает в себя логику, которая итерирует свои <xref:System.Windows.Controls.DockPanel.Dock%2A?displayProperty=nameWithType>элементы ребенка и специально проверяет каждый элемент на определенное значение. Если значение обнаруживается, это значение используется в процессе разметки для расположения дочерних элементов. Использование прилагаемого <xref:System.Windows.Controls.DockPanel.Dock%2A?displayProperty=nameWithType> свойства и этой возможности позиционирования на <xref:System.Windows.Controls.DockPanel> самом деле является мотивирующим сценарием для класса.

## <a name="base-types-and-xaml"></a>Базовые типы и XAML

В основе WPF XAML и его пространства имен XAML лежит коллекция типов, которые соответствуют объектам CLR в дополнение к элементам разметки для XAML. Однако не все классы могут сопоставляться с элементами. Для наследования <xref:System.Windows.Controls.Primitives.ButtonBase>в модели объектов CLR используются абстрактные классы, такие как и некоторые неабстрактные базовые классы. Базовые классы, включая абстрактные, по-прежнему важны для разработки XAML, так как каждый из элементов XAML наследует члены некоторого базового класса в своей иерархии. Часто эти члены включают в себя свойства, которые можно задать в качестве атрибутов в элементе или событий, которые могут быть обработаны. <xref:System.Windows.FrameworkElement>является конкретной базой UI класса WPF на рамочном уровне WPF. При проектировании uI, вы будете использовать различные формы, панели, декоратор, или контроль классов, которые все вытекают из <xref:System.Windows.FrameworkElement>. Связанный базовый <xref:System.Windows.FrameworkContentElement>класс поддерживает элементы, ориентированные на документы, которые хорошо работают для <xref:System.Windows.FrameworkElement>представления макета потока, используя AIS, которые намеренно отражают AA в . Сочетание атрибутов на уровне элемента и модели объекта CLR предоставляет вам набор общих свойств, которые установлены на большинстве конкретных элементов XAML, независимо от конкретного элемента XAML и его основного типа.

## <a name="xaml-security"></a>Безопасность XAML

XAML — это язык разметки, непосредственно представляющий создание и выполнение объекта. Таким образом, элементы, созданные в XAML, имеют такие же возможности взаимодействия с системными ресурсами (например, сетевой доступ и операции ввода-вывода файловой системы), как и эквивалентный созданный код. XAML, загруженный в полностью надежное приложение, имеет тот же доступ к системным ресурсам, что и приложение хостинга.

### <a name="code-access-security-cas-in-wpf"></a>Безопасность доступа к коду (CAS) в WPF

**Этот раздел относится только к рамочной системе .NET. WPF для .NET Core не поддерживает CAS. Для получения дополнительной [Code Access Security differences](../migration/differences-from-net-framework.md#code-access-security)информации см.**

WPF для .NET Framework поддерживает Code Access Security (CAS). Это означает, что содержимое WPF, работая в интернет-зоне, сократило количество разрешений на выполнение. "Loose XAML" (страницы некомпилированного XAML, интерпретируемых во время загрузки xAML viewer) и XBAP обычно работают в этой интернет-зоне и используют тот же набор разрешений. Однако XAML, загруженный в полностью доверенное приложение, имеет такой же доступ к системным ресурсам, как ведущее приложение. Дополнительные сведения см. в разделе [Безопасность частичного доверия в WPF](../../framework/wpf/wpf-partial-trust-security.md).

## <a name="loading-xaml-from-code"></a>Загрузка XAML из кода

XAML можно использовать для определения всего пользовательского интерфейса, но иногда также уместно определение в XAML только его части. Эта возможность может использоваться для включения частичной настройки, локального хранения данных, использования XAML для предоставления бизнес-объекта или различных возможных сценариев. Ключом к этим <xref:System.Windows.Markup.XamlReader> сценариям <xref:System.Windows.Markup.XamlReader.Load%2A> является класс и его метод. Входные данные являются файлом XAML, а выходные данные — объектом, представляющим дерево объектов времени выполнения, созданных по разметке. Затем можно вставить объект в собственность другого объекта, который уже существует в приложении. До тех пор, пока свойство является соответствующим свойством в модели содержимого, которая имеет возможные возможности отображения и которая уведомит движок исполнения о том, что новое содержимое было добавлено в приложение, вы можете легко изменить содержимое запущенного приложения, загрузив его в XAML. Для .NET Framework эта возможность, как правило, доступна только в приложениях с полным доверием из-за очевидных последствий загрузки файлов в приложения по мере их выполнения.
