### <a name="deadlocks"></a><span data-ttu-id="99423-110">Взаимоблокировки</span><span class="sxs-lookup"><span data-stu-id="99423-110">Deadlocks</span></span>  
 <span data-ttu-id="99423-111">Взаимоблокировка происходит, когда каждый из двух потоков пытается заблокировать ресурс, уже заблокированный другим потоком.</span><span class="sxs-lookup"><span data-stu-id="99423-111">A deadlock occurs when each of two threads tries to lock a resource the other has already locked.</span></span> <span data-ttu-id="99423-112">Ни один из потоков не может продолжить работу.</span><span class="sxs-lookup"><span data-stu-id="99423-112">Neither thread can make any further progress.</span></span>  
  
 <span data-ttu-id="99423-113">Многие методы классов управляемых потоков предоставляют значения времени ожидания для обнаружения взаимоблокировок.</span><span class="sxs-lookup"><span data-stu-id="99423-113">Many methods of the managed threading classes provide time-outs to help you detect deadlocks.</span></span> <span data-ttu-id="99423-114">Например, следующий код пытается получить блокировку для объекта с именем `lockObject`.</span><span class="sxs-lookup"><span data-stu-id="99423-114">For example, the following code attempts to acquire a lock on an object named `lockObject`.</span></span> <span data-ttu-id="99423-115">Если блокировка не будет получена в течение 300 миллисекунд, <xref:System.Threading.Monitor.TryEnter%2A?displayProperty=nameWithType> возвратит `false`.</span><span class="sxs-lookup"><span data-stu-id="99423-115">If the lock is not obtained in 300 milliseconds, <xref:System.Threading.Monitor.TryEnter%2A?displayProperty=nameWithType> returns `false`.</span></span>  
  
```vb  
If Monitor.TryEnter(lockObject, 300) Then  
    Try  
        ' Place code protected by the Monitor here.  
    Finally  
        Monitor.Exit(lockObject)  
    End Try  
Else  
    ' Code to execute if the attempt times out.  
End If  
```  
  
```csharp  
if (Monitor.TryEnter(lockObject, 300)) {  
    try {  
        // Place code protected by the Monitor here.  
    }  
    finally {  
        Monitor.Exit(lockObject);  
    }  
}  
else {  
    // Code to execute if the attempt times out.  
}  
```  
  
### <a name="race-conditions"></a><span data-ttu-id="99423-116">Состояние гонки</span><span class="sxs-lookup"><span data-stu-id="99423-116">Race conditions</span></span>  
 <span data-ttu-id="99423-117">Конфликт — это ошибка, которая возникает, когда результат программы зависит от того, какой из двух или более потоков первым достигнет определенного блока кода.</span><span class="sxs-lookup"><span data-stu-id="99423-117">A race condition is a bug that occurs when the outcome of a program depends on which of two or more threads reaches a particular block of code first.</span></span> <span data-ttu-id="99423-118">Выполнение программы часто дает различные результаты, и предсказать результат выполнения конкретного запуска невозможно.</span><span class="sxs-lookup"><span data-stu-id="99423-118">Running the program many times produces different results, and the result of any given run cannot be predicted.</span></span>  
  
 <span data-ttu-id="99423-119">Простой пример состояния гонки — увеличение поля.</span><span class="sxs-lookup"><span data-stu-id="99423-119">A simple example of a race condition is incrementing a field.</span></span> <span data-ttu-id="99423-120">Предположим, что класс содержит закрытое поле **static** (**Shared** в Visual Basic), которое увеличивается всякий раз при создании класса с помощью кода, например `objCt++;` (в C#) или `objCt += 1` (в Visual Basic).</span><span class="sxs-lookup"><span data-stu-id="99423-120">Suppose a class has a private **static** field (**Shared** in Visual Basic) that is incremented every time an instance of the class is created, using code such as `objCt++;` (C#) or `objCt += 1` (Visual Basic).</span></span> <span data-ttu-id="99423-121">Для этой операции необходимо загрузить значение из `objCt` в регистр, увеличить или уменьшить это значение и сохранить его в `objCt`.</span><span class="sxs-lookup"><span data-stu-id="99423-121">This operation requires loading the value from `objCt` into a register, incrementing the value, and storing it in `objCt`.</span></span>  
  
 <span data-ttu-id="99423-122">В многопоточных приложениях поток, загружающий и увеличивающий значение, может быть вытеснен другим потоком, который выполняет все три эти действия; если первый поток возобновляет выполнение и сохраняет его значение, он переопределяет `objCt`, не принимая во внимание тот факт, что в промежутке значение изменилось.</span><span class="sxs-lookup"><span data-stu-id="99423-122">In a multithreaded application, a thread that has loaded and incremented the value might be preempted by another thread which performs all three steps; when the first thread resumes execution and stores its value, it overwrites `objCt` without taking into account the fact that the value has changed in the interim.</span></span>  
  
 <span data-ttu-id="99423-123">Конкретно этого состояния гонки можно легко избежать, применяя методы класса <xref:System.Threading.Interlocked>, например <xref:System.Threading.Interlocked.Increment%2A?displayProperty=nameWithType>.</span><span class="sxs-lookup"><span data-stu-id="99423-123">This particular race condition is easily avoided by using methods of the <xref:System.Threading.Interlocked> class, such as <xref:System.Threading.Interlocked.Increment%2A?displayProperty=nameWithType>.</span></span> <span data-ttu-id="99423-124">Сведения о других технологиях синхронизации данных между несколькими потоками см. в разделе [Синхронизация данных для многопоточности](../../../docs/standard/threading/synchronizing-data-for-multithreading.md).</span><span class="sxs-lookup"><span data-stu-id="99423-124">To read about other techniques for synchronizing data among multiple threads, see [Synchronizing Data for Multithreading](../../../docs/standard/threading/synchronizing-data-for-multithreading.md).</span></span>  
  
 <span data-ttu-id="99423-125">Конфликты могут также возникать при синхронизации действий различных потоков.</span><span class="sxs-lookup"><span data-stu-id="99423-125">Race conditions can also occur when you synchronize the activities of multiple threads.</span></span> <span data-ttu-id="99423-126">При написании каждой строки кода необходимо учитывать, что может произойти, если поток будет вытеснен другим потоком до ее выполнения (или до одной из индивидуальных машинных команд, составляющих эту строку).</span><span class="sxs-lookup"><span data-stu-id="99423-126">Whenever you write a line of code, you must consider what might happen if a thread were preempted before executing the line (or before any of the individual machine instructions that make up the line), and another thread overtook it.</span></span>  
  
## <a name="static-members-and-static-constructors"></a><span data-ttu-id="99423-127">Статические члены и статические конструкторы</span><span class="sxs-lookup"><span data-stu-id="99423-127">Static members and static constructors</span></span>  
 <span data-ttu-id="99423-128">Класс не инициализируется, пока не завершится выполнение его конструктора (конструктор `static` в C# `Shared Sub New` в Visual Basic).</span><span class="sxs-lookup"><span data-stu-id="99423-128">A class is not initialized until its class constructor (`static` constructor in C#, `Shared Sub New` in Visual Basic) has finished running.</span></span> <span data-ttu-id="99423-129">Чтобы предотвратить выполнение кода в еще не инициализированном типе, CLR блокирует все вызовы из других потоков для членов класса `static` (члены `Shared` в Visual Basic) до тех пор, пока выполнение конструктора класса не будет завершено.</span><span class="sxs-lookup"><span data-stu-id="99423-129">To prevent the execution of code on a type that is not initialized, the common language runtime blocks all calls from other threads to `static` members of the class (`Shared` members in Visual Basic) until the class constructor has finished running.</span></span>  
  
 <span data-ttu-id="99423-130">Например, если конструктор класса запускает новый поток, а процедура потока вызывает член `static` класса, новый поток блокируется до завершения конструктора класса.</span><span class="sxs-lookup"><span data-stu-id="99423-130">For example, if a class constructor starts a new thread, and the thread procedure calls a `static` member of the class, the new thread blocks until the class constructor completes.</span></span>

## <a name="general-recommendations"></a><span data-ttu-id="99423-136">Общие рекомендации</span><span class="sxs-lookup"><span data-stu-id="99423-136">General recommendations</span></span>

## <a name="recommendations-for-class-libraries"></a><span data-ttu-id="99423-171">Рекомендации для библиотек классов</span><span class="sxs-lookup"><span data-stu-id="99423-171">Recommendations for class libraries</span></span>  
 <span data-ttu-id="99423-172">При разработке библиотек классов для многопоточности необходимо учитывать следующие рекомендации.</span><span class="sxs-lookup"><span data-stu-id="99423-172">Consider the following guidelines when designing class libraries for multithreading:</span></span>

- <span data-ttu-id="99423-178">Данные экземпляров не должны быть потокобезопасными по умолчанию.</span><span class="sxs-lookup"><span data-stu-id="99423-178">Do not make instance data thread safe by default.</span></span> <span data-ttu-id="99423-179">Добавление блокировок для создания потокобезопасного кода снижает производительность, увеличивает конфликт блокировки и создает условия для возникновения взаимоблокировок.</span><span class="sxs-lookup"><span data-stu-id="99423-179">Adding locks to create thread-safe code decreases performance, increases lock contention, and creates the possibility for deadlocks to occur.</span></span> <span data-ttu-id="99423-180">В обычных моделях приложений пользовательский код одновременно выполняется только одним потоком, что уменьшает необходимость потокобезопасности.</span><span class="sxs-lookup"><span data-stu-id="99423-180">In common application models, only one thread at a time executes user code, which minimizes the need for thread safety.</span></span> <span data-ttu-id="99423-181">По этой причине библиотеки классов .NET Framework не являются потокобезопасными по умолчанию.</span><span class="sxs-lookup"><span data-stu-id="99423-181">For this reason, the .NET Framework class libraries are not thread safe by default.</span></span>
