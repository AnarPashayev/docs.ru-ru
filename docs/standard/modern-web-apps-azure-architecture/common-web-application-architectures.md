## <a name="what-are-layers"></a><span data-ttu-id="e9b4d-132">Что представляют собой слои?</span><span class="sxs-lookup"><span data-stu-id="e9b4d-132">What are layers?</span></span>

<span data-ttu-id="e9b4d-133">По мере увеличения сложности приложения для эффективного управления им может применяться разбиение по обязанностям и задачам.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-133">As applications grow in complexity, one way to manage that complexity is to break up the application according to its responsibilities or concerns.</span></span> <span data-ttu-id="e9b4d-134">Такой подход соответствует принципу разделения задач и помогает сохранить организацию расширяющейся базы кода, благодаря чему разработчики могут легко найти, где именно были реализованы конкретные функциональные возможности.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-134">This follows the separation of concerns principle, and can help keep a growing codebase organized so that developers can easily find where certain functionality is implemented.</span></span> <span data-ttu-id="e9b4d-135">Многослойная архитектура имеет также целый ряд других преимуществ.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-135">Layered architecture offers a number of advantages beyond just code organization, though.</span></span>

<span data-ttu-id="e9b4d-136">Благодаря упорядочению кода с помощью слоев общие низкоуровневые функции могут многократно использоваться по всему приложению.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-136">By organizing code into layers, common low-level functionality can be reused throughout the application.</span></span> <span data-ttu-id="e9b4d-137">Это крайне важно, поскольку такой подход требует меньшего объема кода и, за счет стандартизации приложения на уровне одной реализации, соответствует принципу ["Не повторяйся"](https://en.wikipedia.org/wiki/Don%27t_repeat_yourself).</span><span class="sxs-lookup"><span data-stu-id="e9b4d-137">This reuse is beneficial because it means less code needs to be written and because it can allow the application to standardize on a single implementation, following the [don't repeat yourself (DRY)](https://en.wikipedia.org/wiki/Don%27t_repeat_yourself) principle.</span></span>

<span data-ttu-id="e9b4d-138">В приложениях с многослойной архитектурой могут устанавливаться ограничения на взаимодействие между слоями.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-138">With a layered architecture, applications can enforce restrictions on which layers can communicate with other layers.</span></span> <span data-ttu-id="e9b4d-139">Таким образом удается реализовать инкапсуляцию.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-139">This helps to achieve encapsulation.</span></span> <span data-ttu-id="e9b4d-140">При изменении или замене слоя будут затронуты только те слои, которые работают непосредственно с ним.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-140">When a layer is changed or replaced, only those layers that work with it should be impacted.</span></span> <span data-ttu-id="e9b4d-141">Ограничивая зависимости слоев друг от друга, можно уменьшить последствия внесения изменений, в результате чего единичное изменение не будет влиять на все приложение.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-141">By limiting which layers depend on which other layers, the impact of changes can be mitigated so that a single change doesn't impact the entire application.</span></span>

<span data-ttu-id="e9b4d-142">Применение слоев (и инкапсуляция) позволяет заметно упростить замену функциональных возможностей в рамках приложения.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-142">Layers (and encapsulation) make it much easier to replace functionality within the application.</span></span> <span data-ttu-id="e9b4d-143">Например, приложение может изначально использовать собственную базу данных SQL Server для сохраняемости, а впоследствии перейти на стратегию сохранения состояния на основе облака или веб-API.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-143">For example, an application might initially use its own SQL Server database for persistence, but later could choose to use a cloud-based persistence strategy, or one behind a web API.</span></span> <span data-ttu-id="e9b4d-144">Если в приложении надлежащим образом инкапсулирована реализация сохраняемости на логическом слое, этот слой SQL Server может быть заменен новым, где будет реализовываться тот же открытый интерфейс.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-144">If the application has properly encapsulated its persistence implementation within a logical layer, that SQL Server specific layer could be replaced by a new one implementing the same public interface.</span></span>

<span data-ttu-id="e9b4d-145">Помимо возможности замены реализаций в связи с последующими изменениями, применение слоев в приложении также позволяет менять реализации в целях тестирования.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-145">In addition to the potential of swapping out implementations in response to future changes in requirements, application layers can also make it easier to swap out implementations for testing purposes.</span></span> <span data-ttu-id="e9b4d-146">Вместо написания тестов, которые применяются к слоям реальных данных или пользовательского интерфейса приложения, во время тестирования они заменяются фиктивными реализациями, которые демонстрируют известную реакцию на запросы.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-146">Instead of having to write tests that operate against the real data layer or UI layer of the application, these layers can be replaced at test time with fake implementations that provide known responses to requests.</span></span> <span data-ttu-id="e9b4d-147">Как правило, это значительно упрощает написание и ускоряет выполнение тестов по сравнению с тестированием в реальной инфраструктуре приложения.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-147">This typically makes tests much easier to write and much faster to run when compared to running tests against the application's real infrastructure.</span></span>

<span data-ttu-id="e9b4d-148">Разделение на логические слои широко распространено и помогает упорядочить код приложений предприятия. Сделать это можно несколькими способами.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-148">Logical layering is a common technique for improving the organization of code in enterprise software applications, and there are several ways in which code can be organized into layers.</span></span>

 > <span data-ttu-id="e9b4d-149">_Слои_ обеспечивают логический уровень разделения в приложении.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-149">_Layers_ represent logical separation within the application.</span></span> <span data-ttu-id="e9b4d-150">Если логика приложения физически распределена между несколькими серверами или процессами, такие раздельные физические целевые объекты развертывания называются _уровнями_.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-150">In the event that application logic is physically distributed to separate servers or processes, these separate physical deployment targets are referred to as _tiers_.</span></span> <span data-ttu-id="e9b4d-151">Таким образом, не только возможно, но и широко распространено развертывание N-слойных приложений на одном уровне.</span><span class="sxs-lookup"><span data-stu-id="e9b4d-151">It's possible, and quite common, to have an N-Layer application that is deployed to a single tier.</span></span>
