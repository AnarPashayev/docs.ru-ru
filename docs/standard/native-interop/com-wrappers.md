<span data-ttu-id="67c30-103">Модель COM имеет несколько важных отличий от объектной модели среды выполнения .NET:</span><span class="sxs-lookup"><span data-stu-id="67c30-103">COM differs from the .NET runtime object model in several important ways:</span></span>  
  
- <span data-ttu-id="67c30-104">Клиенты COM-объектов должны управлять временем существования этих объектов. В общеязыковой среде выполнения управление временем существования объектов осуществляет сама среда.</span><span class="sxs-lookup"><span data-stu-id="67c30-104">Clients of COM objects must manage the lifetime of those objects; the common language runtime manages the lifetime of objects in its environment.</span></span>  
  
- <span data-ttu-id="67c30-105">Клиенты COM-объектов определяют доступность службы, запрашивая интерфейс, который ее предоставляет, в результате чего возвращается или не возвращается указатель на интерфейс.</span><span class="sxs-lookup"><span data-stu-id="67c30-105">Clients of COM objects discover whether a service is available by requesting an interface that provides that service and getting back an interface pointer, or not.</span></span> <span data-ttu-id="67c30-106">Клиенты объектов .NET могут получать описание функциональных возможностей объекта с помощью отражения.</span><span class="sxs-lookup"><span data-stu-id="67c30-106">Clients of .NET objects can obtain a description of an object's functionality using reflection.</span></span>  
  
- <span data-ttu-id="67c30-107">Объекты .NET располагаются в памяти под управлением среды выполнения .NET.</span><span class="sxs-lookup"><span data-stu-id="67c30-107">NET objects reside in memory managed by the .NET runtime execution environment.</span></span> <span data-ttu-id="67c30-108">Среда выполнения может перемещать объекты в памяти в целях оптимизации производительности и обновлять все ссылки на перемещаемые объекты.</span><span class="sxs-lookup"><span data-stu-id="67c30-108">The execution environment can move objects around in memory for performance reasons and update all references to the objects it moves.</span></span> <span data-ttu-id="67c30-109">Неуправляемые клиенты после получения указателя на объект работают с объектом, находящимся в указанном расположении.</span><span class="sxs-lookup"><span data-stu-id="67c30-109">Unmanaged clients, having obtained a pointer to an object, rely on the object to remain at the same location.</span></span> <span data-ttu-id="67c30-110">У этих клиентов отсутствуют механизмы для работы с объектами, не имеющими фиксированного расположения.</span><span class="sxs-lookup"><span data-stu-id="67c30-110">These clients have no mechanism for dealing with an object whose location is not fixed.</span></span>  
  
 <span data-ttu-id="67c30-111">Чтобы обойти эти различия, среда выполнения предоставляет классы-оболочки, которые моделируют вызов объектов в соответствующих им средах для управляемых и неуправляемых the runtime creates a [runtime callable wrapper](runtime-callable-wrapper.md) (RCW).</span></span> <span data-ttu-id="67c30-113">Помимо прочего, вызываемая оболочка времени выполнения позволяет следующем рисунке, класс-оболочка, который создается средой выполнения, зависит от контекста вызывающего кода.</span><span class="sxs-lookup"><span data-stu-id="67c30-115">As the following
