# <a name="general-naming-conventions"></a>Общие соглашения об именовании

В этом разделе описываются общие конвенции именования, относящиеся к выбору слов, руководящие принципы использования сокращений и сокращений, а также рекомендации о том, как избежать использования языковых имен.

## <a name="word-choice"></a>Выбор слов
 ✔️ выбрать легко читаемые имена идентификаторов.

 Например, свойство `HorizontalAlignment` с именем более `AlignmentHorizontal`читаемо на английском языке, чем .

 ✔️ do пользу читаемости над краткостью.

 Название `CanScrollHorizontally` свойства лучше, `ScrollableX` чем (неясная ссылка на X-оси).

 ❌НЕ используйте подчеркивания, дефисы или любые другие неальфанумерические символы.

 ❌НЕ используйте венгерскую нотацию.

 ❌AVOID использует идентификаторы, противоречат ключевым словам широко используемых языков программирования.

 Согласно правилу 4 спецификации общего языка (CLS), все совместимые языки должны обеспечить механизм, позволяющий получить доступ к названным элементам, которые используют ключевое слово этого языка в качестве идентификатора. Например, в этом случае знак «К» использует знак «К» в качестве механизма побега. Тем не менее, это все еще хорошая идея, чтобы избежать общих ключевых слов, потому что это гораздо труднее использовать метод с последовательностью побега, чем один без него.

## <a name="using-abbreviations-and-acronyms"></a>Использование аббревиатур и аббревиатур
 ❌НЕ используйте аббревиативы или сокращения как часть имен идентификаторов.

 Например, `GetWindow` использовать, `GetWin`а не .

 ❌НЕ используйте любые аббревиатуры, которые не являются широко принятыми, и даже если они есть, только тогда, когда это необходимо.

## <a name="avoiding-language-specific-names"></a>Избегая языковых конкретных имен
 ✔️ использовать семантически интересные имена, а не язык конкретных ключевых слов для имен типов.

 Например, `GetLength` это имя `GetInt`лучше, чем .

 ✔️ использовать общее имя типа CLR, а не конкретное имя, в тех случаях, когда идентификатор не имеет смыслового значения за пределами своего типа.

 Например, метод преобразования <xref:System.Int64> должны быть `ToInt64`названы , а не `ToLong` (потому что <xref:System.Int64> это `long`имя CLR для C -специфического псевдонима ). В следующей таблице представлено несколько базовых типов данных с использованием имен типа CLR (а также соответствующие имена типов для C, Visual Basic и C q).

|C#|Visual Basic|C++|CLR|
|---------|------------------|-----------|---------|
|**sbyte**|**SByte**|**char**|**SByte**|
|**byte**|**Byte**|**unsigned char**|**Byte**|
|**short**|**Короткие**|**short**|**Int16**|
|**ushort**|**UInt16**|**unsigned short**|**UInt16**|
|**Int**|**Целое число**|**Int**|**Int32**|
|**uint**|**UInt32**|**unsigned int**|**UInt32**|
|**Длинные**|**Длинные**|**__int64**|**Int64**|
|**Ulong**|**UInt64**|**unsigned __int64**|**UInt64**|
|**Плавать**|**Single**|**Плавать**|**Single**|
|**double**|**Double**|**double**|**Double**|
|**bool**|**Boolean**|**bool**|**Boolean**|
|**char**|**Char**|**wchar_t**|**Char**|
|**строка**|**String**|**String**|**String**|
|**Объекта**|**Объект**|**Объект**|**Объект**|

 ✔️ использовать общее имя, `value` например, или, `item`вместо того, чтобы повторять имя типа, в тех редких случаях, когда идентификатор не имеет смыслового значения и тип параметра не имеет значения.

## <a name="naming-new-versions-of-existing-apis"></a>Наименование новых версий существующих AIS
 ✔️ использовать имя, похожее на старый API при создании новых версий существующего API.

 Это помогает подчеркнуть взаимосвязь между AIS.

 ✔️ предпочитают добавлять суффикс, а не префикс, чтобы указать новую версию существующего API.

 Это поможет обнаружить при просмотре документации или с помощью IntelliSense. Старая версия API будет организована рядом с новыми API, потому что большинство браузеров и IntelliSense показывают идентификаторы в алфавитном порядке.

 ✔️ CONSIDER с помощью совершенно нового, но значимого идентификатора, вместо добавления суффикса или префикса.

 ✔️ do использовать числовой суффикс, чтобы указать новую версию существующего API, особенно если существующее название API является единственным именем, которое имеет смысл (т.е. если это отраслевой стандарт), и если добавление какого-либо значимого суффикса (или изменение названия) не является подходящим вариантом.

 ❌НЕ используйте суффикс "Ex" (или аналогичный) для идентификатора, чтобы отличить его от более ранней версии того же API.

 ✔️ использовать суффикс "64" при внедрении версий AIS, которые работают на 64-битный целый ряд (длинный целый ряд) вместо 32-битного целых. Этот подход необходимо подходить только при наличии существующего 32-битного API; не делайте это для совершенно новых AIS только с 64-битной версией.
