# <a name="base-classes-for-implementing-abstractions"></a>Базовые классы для реализации абстракций
Строго говоря, класс становится базовым классом, если другой класс является производным от него. Однако в этом разделе базовый класс является классом, предназначенным главным образом для предоставления общей абстракции или другим классам для повторного использования некоторой реализации по умолчанию, хотя наследование. Базовые классы обычно располагаются в середине иерархий наследования между абстракцией в корне иерархии и несколькими пользовательскими реализациями в нижней части.  
  
 Они служат в качестве вспомогательных методов реализации для реализации абстракций. Например, одна из абстракций платформы для упорядоченных коллекций элементов — это интерфейс <xref:System.Collections.Generic.IList%601>. Реализация <xref:System.Collections.Generic.IList%601> не является тривиальной, и поэтому платформа предоставляет несколько базовых классов, таких как <xref:System.Collections.ObjectModel.Collection%601> и <xref:System.Collections.ObjectModel.KeyedCollection%602>, которые служат вспомогательными методами для реализации пользовательских коллекций.  
  
 Базовые классы обычно не подходят для использования в качестве абстракций, так как они обычно содержат слишком много реализации. Например, базовый класс `Collection<T>` содержит множество реализаций, связанных с тем фактом, что он реализует неуниверсальный `IList` интерфейс (для лучшей интеграции с неуниверсальными коллекциями) и тот факт, что это коллекция элементов, хранящихся в памяти в одном из своих полей.  
  
 Как уже говорилось, базовые классы могут предоставить бесценную помощь пользователям, которым необходимо реализовать абстракции, но в то же время они могут быть значительной ответственностью. Они добавляют контактную зону и увеличивают глубину иерархий наследования и поэтому концептуально усложняют платформу. Поэтому базовые классы следует использовать только в том случае, если они предоставляют значительные преимущества пользователям платформы. Их следует избегать, если они предоставляют значение только разработчикам платформы, в этом случае следует строго рассматривать делегирование внутренней реализации, а не наследование от базового класса.  
  
 **✓ CONSIDER** внесения базовые классы абстрактными, даже если они не содержат абстрактных членов. Это явно сообщает пользователям, что класс предназначен только для наследования.  
  
 **✓ CONSIDER** размещать базовые классы в отдельном пространстве имен типов основным сценарием. Базовые классы по определению предназначены для расширенных сценариев расширяемости, поэтому они не интересны большинству пользователей.  
  
 **X AVOID** именования базовых классов с суффиксом «Базовый», если класс предназначен для использования в открытых интерфейсах API.
