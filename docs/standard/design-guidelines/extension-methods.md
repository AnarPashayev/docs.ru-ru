# <a name="extension-methods"></a>Методы расширения
Методы расширения являются компонентом языка, который позволяет статических методов, вызываемых с помощью синтаксиса вызова метода экземпляра. Эти методы необходимо выполнить хотя бы один параметр, который представляет экземпляр, метод должен работать.  
  
 Класс, который определяет такие методы расширения упоминается как класса «спонсора», и они должны объявляться как статические. Чтобы использовать методы расширения, один необходимо импортировать пространство имен определения класса спонсора.  
  
 **X AVOID** frivolously определяет методы расширения, особенно на типах, вы не являетесь его владельцем.  
  
 Если у вас есть исходный код типа, рекомендуется использовать методы регулярных экземпляра. Если вы не являетесь, и вы хотите добавить метод, необходимо Будьте очень осторожным. Если правильно использовать методы расширения может загромождения интерфейсов API типов, которые не были предназначены для этих методов.  
  
 **✓ CONSIDER** с помощью методов расширения в следующих сценариях:  
  
- Для предоставления вспомогательные функции, необходимые для каждой реализации интерфейса, если говорят, что функциональные возможности могут быть написаны с точки зрения основной интерфейс. Это обусловлено конкретные реализации в противном случае не могут быть назначены интерфейсов. Например `LINQ to Objects` операторы реализуются как методы расширения для всех <xref:System.Collections.Generic.IEnumerable%601> типов. Таким образом, любой `IEnumerable<>` реализуется автоматически с поддержкой LINQ.  
  
- Когда метод экземпляра стало бы причиной возникновения зависит от какого-либо типа, но такая зависимость будет нарушена правил управления зависимостей. К примеру, зависимость от <xref:System.String> для <xref:System.Uri?displayProperty=nameWithType> скорее всего, не является предпочтительным и поэтому `String.ToUri()` экземпляр метода возвращает `System.Uri` бы использования неправильного проекта с точки зрения управления зависимостей. Статический метод расширения `Uri.ToUri(this string str)` возвращение `System.Uri` было бы гораздо лучше.  
  
 **X AVOID** определения методов расширения в <xref:System.Object?displayProperty=nameWithType>.  
  
 VB пользователи не будут иметь возможность вызывать такие методы для ссылки на объекты, с помощью синтаксиса метода расширения. VB не поддерживает вызов таких методов, так как для VB, объявления ссылки, как объект заставляет все вызовы методов в нем опаздываете привязка к (фактический элемент с именем определяется во время выполнения), во время привязки к методам расширения определяются во время компиляции (ранняя привязки).  
  
 Обратите внимание на то, что правило применяется для других языков, где присутствует одно и то же поведение привязки, или где методы расширения не поддерживаются.  
  
 **X DO NOT** размещения методов расширения в пространстве имен, расширенного типа, если он предназначен для добавления методов в интерфейсах или для управления зависимостями.  
  
 **X AVOID** определение два или несколько методов расширения с такой же сигнатурой, даже если они находятся в разных пространствах имен.  
  
 **✓ CONSIDER** определение методов расширения в пространстве имен, расширенный тип, если тип является интерфейсом и методы расширения предназначены для использования в большинство или все случаи.  
  
 **X DO NOT** определяет расширение методов реализации функции в пространствах имен, обычно связанные с другими компонентами. Вместо этого их определения в пространстве имен, связанного с данным компонентом, которому они принадлежат.  
  
 **X AVOID** универсального именования пространств имен, выделенных для методов расширения (например, «расширения»). Используйте описательное имя (например, «маршрутизации») вместо этого.
