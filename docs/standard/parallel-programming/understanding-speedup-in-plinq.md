# <a name="understanding-speedup-in-plinq"></a><span data-ttu-id="11af9-102">Общее представление об ускорении выполнения в PLINQ</span><span class="sxs-lookup"><span data-stu-id="11af9-102">Understanding Speedup in PLINQ</span></span>
<span data-ttu-id="11af9-103">Основное назначение PLINQ — ускорять обработку запросов LINQ to Objects на компьютерах с многоядерными процессорами, выполняя делегаты запроса в параллельном режиме.</span><span class="sxs-lookup"><span data-stu-id="11af9-103">The primary purpose of PLINQ is to speed up the execution of LINQ to Objects queries by executing the query delegates in parallel on multi-core computers.</span></span> <span data-ttu-id="11af9-104">Преимущества PLINQ проявляются лучше всего, когда обработка каждого элемента в исходной коллекции не зависит от других элементов и отдельные делегаты не используют общее состояние.</span><span class="sxs-lookup"><span data-stu-id="11af9-104">PLINQ performs best when the processing of each element in a source collection is independent, with no shared state involved among the individual delegates.</span></span> <span data-ttu-id="11af9-105">Такие операции достаточно часто встречаются в LINQ to Objects и PLINQ. Они являются *параллельными*, так как легко поддаются разделению на несколько потоков.</span><span class="sxs-lookup"><span data-stu-id="11af9-105">Such operations are common in LINQ to Objects and PLINQ, and are often called "*delightfully parallel*" because they lend themselves easily to scheduling on multiple threads.</span></span> <span data-ttu-id="11af9-106">Но не все запросы состоят из таких параллельных операций. Очень часто запросы содержат некоторые операторы, которые не могут выполняться параллельно или существенно замедляют параллельное выполнение.</span><span class="sxs-lookup"><span data-stu-id="11af9-106">However, not all queries consist entirely of delightfully parallel operations;  <span data-ttu-id="11af9-112">В следующих разделах перечисляются самые важные факторы, которые влияют на производительность параллельного выполнения запросов.</span><span class="sxs-lookup"><span data-stu-id="11af9-112">The following sections lists some of the most important factors that impact parallel query performance.</span></span> <span data-ttu-id="11af9-113">Это инструкции общего характера, которые нельзя использовать для прогнозирования производительности запросов во всех сценариях.</span><span class="sxs-lookup"><span data-stu-id="11af9-113">These are general statements that by themselves are not sufficient to predict query performance in all cases.</span></span> <span data-ttu-id="11af9-114">Для точной оценки нужно измерить фактическую производительность конкретных запросов на конкретных компьютерах в разных сочетаниях репрезентативных конфигураций и нагрузок.</span><span class="sxs-lookup"><span data-stu-id="11af9-114">As always, it is important to measure actual performance of specific queries on computers with a range of representative configurations and loads.</span></span>  
  
1. <span data-ttu-id="11af9-115">Вычислительная стоимость общей работы.</span><span class="sxs-lookup"><span data-stu-id="11af9-115">Computational cost of the overall work.</span></span>  
  
     <span data-ttu-id="11af9-116">Чтобы ускорить обработку запросов PLINQ, нужен достаточный объем параллельных операций, позволяющий компенсировать накладные расходы.</span><span class="sxs-lookup"><span data-stu-id="11af9-116">To achieve speedup, a PLINQ query must have enough delightfully parallel work to offset the overhead.</span></span> <span data-ttu-id="11af9-117">Работу можно оценить как произведение вычислительной стоимости каждого делегата на число элементов в исходной коллекции.</span><span class="sxs-lookup"><span data-stu-id="11af9-117">The work can be expressed as the computational cost of each delegate multiplied by the number of elements in the source collection.</span></span> <span data-ttu-id="11af9-118">Если операция допускает параллелизацию, потенциал ускорения напрямую зависит от ее вычислительной стоимости.</span><span class="sxs-lookup"><span data-stu-id="11af9-118">Assuming that an operation can be parallelized, the more computationally expensive it is, the greater the opportunity for speedup.</span></span> <span data-ttu-id="11af9-119">Например, если функция выполняется за одну миллисекунду, последовательный запрос к 1000 элементов будет выполняться около одной секунды.      <span data-ttu-id="11af9-124">Запрос queryA из следующего примера можно считать хорошим кандидатом для PLINQ, если его функция Select предусматривает много работы.</span><span class="sxs-lookup"><span data-stu-id="11af9-124">In the following example, queryA is probably a good candidate for PLINQ, assuming that its Select function involves a lot of work.</span></span> <span data-ttu-id="11af9-125">Запрос queryB вряд ли хорошо подходит для параллелизации, так как в его инструкции Select выполняется мало работы, и накладные расходы перевесят все возможное ускорение или значительную его часть.</span><span class="sxs-lookup"><span data-stu-id="11af9-125">queryB is probably not a good candidate because there is not enough work in the Select statement, and the overhead of parallelization will offset most or all of the speedup.</span></span>

