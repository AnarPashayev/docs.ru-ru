# <a name="streaming-message-transfer"></a><span data-ttu-id="286db-102">Потоковая передача сообщений</span><span class="sxs-lookup"><span data-stu-id="286db-102">Streaming Message Transfer</span></span>

<span data-ttu-id="286db-103">Транспорты Windows Communication Foundation (WCF) поддерживают два режима передачи сообщений:</span><span class="sxs-lookup"><span data-stu-id="286db-103">Windows Communication Foundation (WCF) transports support two modes for transferring messages:</span></span>

- <span data-ttu-id="286db-106">При потоковой передаче сообщение передается в виде потока.</span><span class="sxs-lookup"><span data-stu-id="286db-106">Streamed transfers expose the message as a stream.</span></span> <span data-ttu-id="286db-107">Получатель начинает обработку сообщения до того, как оно будет полностью доставлено.</span><span class="sxs-lookup"><span data-stu-id="286db-107">The receiver starts processing the message before it is completely delivered.</span></span>  
  
- <span data-ttu-id="286db-108">Потоковая передача позволяет улучшить возможности масштабирования службы за счет исключения необходимости использовать большие буферы памяти.</span><span class="sxs-lookup"><span data-stu-id="286db-108">Streamed transfers can improve the scalability of a service by eliminating the requirement for large memory buffers.</span></span> <span data-ttu-id="286db-109">Степень улучшения масштабируемости при изменении режима передачи зависит от размера передаваемых сообщений.</span><span class="sxs-lookup"><span data-stu-id="286db-109">Whether changing the transfer mode improves scalability depends on the size of the messages being transferred.</span></span> <span data-ttu-id="286db-110">Для сообщений большого размера предпочтительнее использовать потоковую передачу.</span><span class="sxs-lookup"><span data-stu-id="286db-110">Large message sizes favor using streamed transfers.</span></span>  
  
 <span data-ttu-id="286db-111">По умолчанию транспорты HTTP, TCP/IP и именованных каналов используют буферизованную передачу.</span><span class="sxs-lookup"><span data-stu-id="286db-111">By default, the HTTP, TCP/IP, and named pipe transports use buffered transfers.</span></span> <span data-ttu-id="286db-112">Этот документ описывает изменение режима передачи с буферизованной на потоковую для этих транспортов, а также возможные последствия.</span><span class="sxs-lookup"><span data-stu-id="286db-112">This document describes how to switch these transports from a buffered to streamed transfer mode and the consequences of doing so.</span></span>

 <span data-ttu-id="286db-118">Привязки <xref:System.ServiceModel.BasicHttpBinding>, <xref:System.ServiceModel.NetTcpBinding> и <xref:System.ServiceModel.NetNamedPipeBinding> предоставляют свойство <xref:System.ServiceModel.TransferMode>.</span><span class="sxs-lookup"><span data-stu-id="286db-118">The <xref:System.ServiceModel.BasicHttpBinding>, <xref:System.ServiceModel.NetTcpBinding>, and <xref:System.ServiceModel.NetNamedPipeBinding> bindings expose the <xref:System.ServiceModel.TransferMode> property.</span></span> <span data-ttu-id="286db-119">Для других транспортов необходимо создать пользовательскую привязку для задания режима передачи.</span><span class="sxs-lookup"><span data-stu-id="286db-119">For other transports, you must create a custom binding to set the transfer mode.</span></span>  
  
 <span data-ttu-id="286db-120">Решение по использованию буферизованной или потоковой передачи является локальным решением конечной точки.</span><span class="sxs-lookup"><span data-stu-id="286db-120">The decision to use either buffered or streamed transfers is a local decision of the endpoint.</span></span> <span data-ttu-id="286db-121">Для транспортов HTTP режим передачи не распространяется через подключение или на серверы и другие посредники.</span><span class="sxs-lookup"><span data-stu-id="286db-121">For HTTP transports, the transfer mode does not propagate across a connection, or to servers and other intermediaries.</span></span> <span data-ttu-id="286db-122">Выбор режима передачи не отражается в описании интерфейса службы.</span><span class="sxs-lookup"><span data-stu-id="286db-122">Setting the transfer mode is not reflected in the description of the service interface.</span></span> <span data-ttu-id="286db-123">Чтобы задать режим, после создания класса клиента для службы необходимо изменить файл конфигурации для тех служб, которые планируется использовать с потоковой передачей.</span><span class="sxs-lookup"><span data-stu-id="286db-123">After generating a client class for a service, you must edit the configuration file for services intended to be used with streamed transfers to set the mode.</span></span> <span data-ttu-id="286db-124">Для транспортов TCP и именованного канала режим передачи распространяется в виде утверждения политики.</span><span class="sxs-lookup"><span data-stu-id="286db-124">For TCP and named pipe transports, the transfer mode is propagated as a policy assertion.</span></span>

## <a name="enabling-asynchronous-streaming"></a><span data-ttu-id="286db-126">Включение асинхронной потоковой передачи</span><span class="sxs-lookup"><span data-stu-id="286db-126">Enabling Asynchronous Streaming</span></span>  

 <span data-ttu-id="286db-127">Чтобы включить поддержку асинхронной потоковой передачи данных, добавьте поведение конечной точки <xref:System.ServiceModel.Description.DispatcherSynchronizationBehavior> в основное приложение службы и установите свойству <xref:System.ServiceModel.Description.DispatcherSynchronizationBehavior.AsynchronousSendEnabled%2A> значение `true`.</span><span class="sxs-lookup"><span data-stu-id="286db-127">To enable asynchronous streaming, add the  <xref:System.ServiceModel.Description.DispatcherSynchronizationBehavior> endpoint behavior to the service host and set its <xref:System.ServiceModel.Description.DispatcherSynchronizationBehavior.AsynchronousSendEnabled%2A> property to `true`.</span></span>  
  
 <span data-ttu-id="286db-128">Эта версия WCF также добавляет возможность истинной асинхронной потоковой передачи на стороне отправки.</span><span class="sxs-lookup"><span data-stu-id="286db-128">This version of WCF also adde the capability of true asynchronous streaming on the send side.</span></span> <span data-ttu-id="286db-129">Это повышает масштабируемость службы в сценариях, когда обрабатываются потоковые сообщения для нескольких клиентов. ## <a name="restrictions-on-streamed-transfers"></a><span data-ttu-id="286db-132">Ограничения потоковой передачи</span><span class="sxs-lookup"><span data-stu-id="286db-132">Restrictions on Streamed Transfers</span></span>

 <span data-ttu-id="286db-134">Операции, происходящие в потоковом транспорте, могут иметь контракт не более чем с одним входным или выходным параметром.</span><span class="sxs-lookup"><span data-stu-id="286db-134">Operations that occur across a streamed transport can have a contract with at most one input or output parameter.</span></span> <span data-ttu-id="286db-135">Такой параметр соответствует всему телу сообщения и должен принадлежать к типу <xref:System.ServiceModel.Channels.Message>, наследуемому от <xref:System.IO.Stream>, или реализации <xref:System.Xml.Serialization.IXmlSerializable>.</span><span class="sxs-lookup"><span data-stu-id="286db-135">That parameter corresponds to the entire body of the message and must be a <xref:System.ServiceModel.Channels.Message>, a derived type of <xref:System.IO.Stream>, or an <xref:System.Xml.Serialization.IXmlSerializable> implementation.</span></span> <span data-ttu-id="286db-136">Наличие возвращаемого значения операции эквивалентно наличию выходного параметра.</span><span class="sxs-lookup"><span data-stu-id="286db-136">Having a return value for an operation is equivalent to having an output parameter.</span></span>
