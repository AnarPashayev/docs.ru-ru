## <a name="poison-message-handling"></a><span data-ttu-id="09599-117">Обработка подозрительных сообщений</span><span class="sxs-lookup"><span data-stu-id="09599-117">Poison-Message Handling</span></span>  
 <span data-ttu-id="09599-118">Подозрительное сообщение представляет собой сообщение, для которого превышено максимальное количество попыток доставки в принимающее приложение.</span><span class="sxs-lookup"><span data-stu-id="09599-118">A poison message is a message that has exceeded the maximum number of delivery attempts to the receiving application.</span></span> <span data-ttu-id="09599-119">Такая ситуация может возникнуть, когда приложение, считывающее сообщение из транзакционной очереди, не может сразу обработать сообщение из-за ошибок.</span><span class="sxs-lookup"><span data-stu-id="09599-119">This situation can arise when an application that reads a message from a transactional queue cannot process the message immediately because of errors.</span></span> <span data-ttu-id="09599-120">Если приложение отменяет транзакцию, при которой было получено сообщение из очереди, оно возвращает сообщение обратно в очередь.</span><span class="sxs-lookup"><span data-stu-id="09599-120">If the application aborts the transaction in which the queued message was received, it returns the message to the queue.</span></span> <span data-ttu-id="09599-121">После этого приложение пытается повторно извлечь сообщение в новой транзакции.</span><span class="sxs-lookup"><span data-stu-id="09599-121">The application then tries to retrieve the message again in a new transaction.</span></span> <span data-ttu-id="09599-122">Если причина ошибки не устранена, принимающее приложение может зациклиться, получая и прерывая прием одного и того же сообщения, до достижения максимального числа попыток доставки и пометки сообщения как подозрительного.</span><span class="sxs-lookup"><span data-stu-id="09599-122">If the problem that causes the error is not corrected, the receiving application can get stuck in a loop receiving and aborting the same message until it exceeds the maximum number of delivery attempts, and a poison message results.</span></span>  
  
 <span data-ttu-id="09599-123">Основные различия между очередью сообщений (MSMQ) в Windows Vista, [!INCLUDE[ws2003](../../../../includes/ws2003-md.md)]и [!INCLUDE[wxp](../../../../includes/wxp-md.md)], которые относятся к обработке подозрительных событий, включают следующее:</span><span class="sxs-lookup"><span data-stu-id="09599-123">The key differences between Message Queuing (MSMQ) on Windows Vista, [!INCLUDE[ws2003](../../../../includes/ws2003-md.md)], and [!INCLUDE[wxp](../../../../includes/wxp-md.md)] that are relevant to poison handling include the following:</span></span>  
  
- <span data-ttu-id="09599-124">MSMQ в Windows Vista поддерживает подочереди, тогда как [!INCLUDE[ws2003](../../../../includes/ws2003-md.md)] и [!INCLUDE[wxp](../../../../includes/wxp-md.md)] не поддерживают подочереди.</span><span class="sxs-lookup"><span data-stu-id="09599-124">MSMQ in Windows Vista supports subqueues, while [!INCLUDE[ws2003](../../../../includes/ws2003-md.md)] and [!INCLUDE[wxp](../../../../includes/wxp-md.md)] do not support subqueues.</span></span> <span data-ttu-id="09599-125">Вложенные очереди используются при обработке подозрительных сообщений.</span><span class="sxs-lookup"><span data-stu-id="09599-125">Subqueues are used in poison-message handling.</span></span> <span data-ttu-id="09599-126">Очереди повторных попыток и очередь подозрительных сообщений являются вложенными очередями очереди приложения, созданной на основе параметров обработки подозрительных сообщений.</span><span class="sxs-lookup"><span data-stu-id="09599-126">The retry queues and the poison queue are subqueues to the application queue that is created based on the poison-message handling settings.</span></span> <span data-ttu-id="09599-127">Значение `MaxRetryCycles` указывает, сколько необходимо создать вложенных очередей повторных попыток.</span><span class="sxs-lookup"><span data-stu-id="09599-127">The `MaxRetryCycles` dictates how many retry subqueues to create.</span></span> <span data-ttu-id="09599-128">Поэтому при работе с [!INCLUDE[ws2003](../../../../includes/ws2003-md.md)] или [!INCLUDE[wxp](../../../../includes/wxp-md.md)] параметр `MaxRetryCycles` игнорируется, а значение `ReceiveErrorHandling.Move` не допускается.</span><span class="sxs-lookup"><span data-stu-id="09599-128">Therefore, when running on [!INCLUDE[ws2003](../../../../includes/ws2003-md.md)] or [!INCLUDE[wxp](../../../../includes/wxp-md.md)], `MaxRetryCycles` are ignored and `ReceiveErrorHandling.Move` is not allowed.</span></span>  
  
- <span data-ttu-id="09599-129">MSMQ в Windows Vista поддерживает отрицательное подтверждение, а [!INCLUDE[ws2003](../../../../includes/ws2003-md.md)] и [!INCLUDE[wxp](../../../../includes/wxp-md.md)] не поддерживаются.</span><span class="sxs-lookup"><span data-stu-id="09599-129">MSMQ in Windows Vista supports negative acknowledgment, while [!INCLUDE[ws2003](../../../../includes/ws2003-md.md)] and [!INCLUDE[wxp](../../../../includes/wxp-md.md)] do not.</span></span> <span data-ttu-id="09599-130">Уведомление о недоставке от диспетчера принимающей очереди приводит к тому, что диспетчер передающей очереди помещает сообщение в очередь недоставленных сообщений.</span><span class="sxs-lookup"><span data-stu-id="09599-130">A negative acknowledgment from the receiving queue manager causes the sending queue manager to place the rejected message in the dead-letter queue.</span></span> <span data-ttu-id="09599-131">Поэтому значение `ReceiveErrorHandling.Reject` не допускается в [!INCLUDE[ws2003](../../../../includes/ws2003-md.md)] и [!INCLUDE[wxp](../../../../includes/wxp-md.md)].</span><span class="sxs-lookup"><span data-stu-id="09599-131">As such, `ReceiveErrorHandling.Reject` is not allowed with [!INCLUDE[ws2003](../../../../includes/ws2003-md.md)] and [!INCLUDE[wxp](../../../../includes/wxp-md.md)].</span></span>  
  
- <span data-ttu-id="09599-132">MSMQ в Windows Vista поддерживает свойство Message, которое хранит количество попыток доставки сообщений.</span><span class="sxs-lookup"><span data-stu-id="09599-132">MSMQ in Windows Vista supports a message property that keeps count of the number of times message delivery is attempted.</span></span> <span data-ttu-id="09599-133">Это свойство подсчета прекращений недоступно в [!INCLUDE[ws2003](../../../../includes/ws2003-md.md)] и [!INCLUDE[wxp](../../../../includes/wxp-md.md)].</span><span class="sxs-lookup"><span data-stu-id="09599-133">This abort count property is not available on [!INCLUDE[ws2003](../../../../includes/ws2003-md.md)] and [!INCLUDE[wxp](../../../../includes/wxp-md.md)].</span></span> <span data-ttu-id="09599-134">WCF поддерживает счетчик прерываний в памяти, поэтому возможно, что это свойство не может содержать точное значение, если одно и то же сообщение считывается несколькими службами WCF в веб-ферме.</span><span class="sxs-lookup"><span data-stu-id="09599-134">WCF maintains the abort count in memory, so it is possible that this property may not contain an accurate value when the same message is read by more than one WCF service in a Web farm.</span></span>

 <span data-ttu-id="09599-136">MSMQ в Windows Vista поддерживает удаленные транзакционные операции чтения.</span><span class="sxs-lookup"><span data-stu-id="09599-136">MSMQ on Windows Vista supports remote transactional reads.</span></span> <span data-ttu-id="09599-137">Это позволяет приложению, считывающему из очереди, находиться на компьютере, отличном от компьютера, на котором размещена очередь.</span><span class="sxs-lookup"><span data-stu-id="09599-137">This allows an application that is reading from a queue to be hosted on a computer that is different from the computer on which the queue is hosted.</span></span> <span data-ttu-id="09599-138">Этим обеспечивается возможность использования фермы служб, осуществляющих чтение из центральной очереди, что увеличивает общую производительность системы.</span><span class="sxs-lookup"><span data-stu-id="09599-138">This ensures the ability to have a farm of services reading from a central queue, which increases the overall throughput of the system.</span></span> <span data-ttu-id="09599-139">Кроме того, если во время чтения и обработки сообщения возникает ошибка, такой подход позволяет откатить транзакцию, и сообщение останется в очереди для последующей обработки.</span><span class="sxs-lookup"><span data-stu-id="09599-139">It also ensures that if a failure occurs when reading and processing the message, the transaction rolls back and the message remains in the queue for later processing.</span></span>
