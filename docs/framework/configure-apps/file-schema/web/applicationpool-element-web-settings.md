# <a name="applicationpool-element-web-settings"></a>\<Элемент applicationPool> (веб-параметры)
Отогаляет параметры конфигурации, которые используются ASP.NET для управления поведением в масштабах всего процесса, когда ASP.NET приложение работает в интегрированном режиме на IIS 7.0 или более поздней версии.  
  
> [!IMPORTANT]
> Этот элемент и функция, которую он поддерживает, работают только в том случае, если ваше ASP.NET приложение размещается на IIS 7.0 или более поздних версиях.  
  
[**\<конфигурация>**](../configuration-element.md)  
&nbsp;&nbsp;[**\<system.web>**](system-web-element-web-settings.md)  
&nbsp;&nbsp;&nbsp;&nbsp;**\<applicationPool>**  
  
## <a name="syntax"></a>Синтаксис  
  
```xml  
<applicationPool
    maxConcurrentRequestsPerCPU="5000"
    maxConcurrentThreadsPerCPU="0"
    requestQueueLimit="5000" />  
```  
  
## <a name="attributes-and-elements"></a>Атрибуты и элементы  

В следующих разделах описаны атрибуты, дочерние и родительские элементы.  
  
### <a name="attributes"></a>Атрибуты  
  
|attribute|Описание|  
|---------------|-----------------|  
|`maxConcurrentRequestsPerCPU`|Определяет, сколько одновременных запросов ASP.NET позволяет на процессор.|  
|`maxConcurrentThreadsPerCPU`|Определяет, сколько одновременных потоков может быть запущено для пула приложений для каждого процессора. Это обеспечивает альтернативный способ управления ASP.NET параллелизма, поскольку можно ограничить количество управляемых потоков, которые могут быть использованы на процессоре для обслуживания запросов. По умолчанию эта настройка составляет 0, что означает, что ASP.NET не ограничивает количество потоков, которые могут быть созданы на процессор, хотя пул потоков CLR также ограничивает количество потоков, которые могут быть созданы.|  
|`requestQueueLimit`|Упоняет максимальное количество запросов, которые могут быть в очереди для ASP.NET в одном процессе. Когда два или более ASP.NET приложений работают в одном пуле приложений, совокупный набор запросов, поданных в любое приложение в пуле приложений, подлежит этой настройке.|  
  
### <a name="child-elements"></a>Дочерние элементы  
 Нет.  
  
### <a name="parent-elements"></a>Родительские элементы  
  
|Элемент|Описание|  
|-------------|-----------------|  
|[\<system.web>](system-web-element-web-settings.md)|Содержит информацию о том, как ASP.NET взаимодействует с хост-приложением.|  
  
## <a name="remarks"></a>Remarks  

При запуске IIS 7.0 или более поздней версии в интегрированном режиме эта комбинация элементов позволяет настроить, как ASP.NET управляет потоками и очередями запросов при размещении приложения в пуле приложений IIS. Если вы запустите IIS 6 или вы работаете IIS 7.0 в классическом режиме или в режиме ISAPI, эти параметры игнорируются.  
  
Настройки `applicationPool` применяются ко всем пулам приложений, которые работают на определенной версии рамочного соглашения .NET. Настройки содержатся в файле aspnet.config. Существует версия этого файла для версий 2.0 и 4.0 из .NET Framework. (Версии 3.0 и 3.5 рамочного соглашения .NET делятся файлом aspnet.config с версией 2.0.)  
  
> [!IMPORTANT]
> Если вы работаете IIS 7.0 на Windows 7, вы можете настроить отдельный файл aspnet.config для каждого пула приложений. Это позволяет адаптировать производительность потоков для каждого пула приложений.  
  
Для `maxConcurrentRequestsPerCPU` настройки настройки "5000" по умолчанию в рамках .NET 4 эффективно выключает регулирование запроса, которое контролируется ASP.NET, если только у вас на самом деле нет 5000 или более запросов на процессор. Вместо этого параметр по умолчанию зависит от пула потоков CLR для автоматического управления параллелизмом на процессоре. Приложения, широко используют асинхронную обработку запросов или которые имеют много долгосрочных запросов, заблокированных в ввод-во-от сети, выиграют от увеличения лимита по умолчанию в системе .NET 4. Установка `maxConcurrentRequestsPerCPU` к нулю отключает использование управляемых потоков для обработки ASP.NET запросов. При запуске приложения в пуле приложений IIS запросы остаются на потоке IIS IIS IIS I/O, и поэтому параллелизм регулируется настройками потока IIS.  
  
Настройка `requestQueueLimit` работает так же, как `requestQueueLimit` атрибут элемента [processModel,](https://docs.microsoft.com/previous-versions/dotnet/netframework-4.0/7w2sway1(v=vs.100)) который устанавливается в файлах Web.config для ASP.NET приложений. Однако `requestQueueLimit` настройка в файле aspnet.config переопределяет настройки `requestQueueLimit` в файле Web.config. Другими словами, если оба атрибута установлены (по `requestQueueLimit` умолчанию это верно), то параметр в файле aspnet.config имеет приоритет.  
  
## <a name="example"></a>Пример  

В следующем примере показано, как настроить поведение ASP.NET процесса в файле aspnet.config в следующих обстоятельствах:  
  
- Приложение размещается в пуле приложений IIS 7.0.  
  
- IIS 7.0 работает в интегрированном режиме.  
  
- Приложение использует .NET Framework 3.5 SP1 или более позднюю версию.  
  
Значения в примере являются значениями по умолчанию.  
  
```xml  
<configuration>  
  <system.web>  
    <applicationPool
        maxConcurrentRequestsPerCPU="5000"  
        maxConcurrentThreadsPerCPU="0"
        requestQueueLimit="5000" />  
  </system.web>  
</configuration>  
```
