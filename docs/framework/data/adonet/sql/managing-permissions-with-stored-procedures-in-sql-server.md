<span data-ttu-id="d8801-104">Одним из методов создания нескольких линий защиты для базы данных является реализация доступа ко всем данным с помощью хранимых процедур или определяемых пользователем функций.</span><span class="sxs-lookup"><span data-stu-id="d8801-104">One method of creating multiple lines of defense around your database is to implement all data access using stored procedures or user-defined functions.</span></span> <span data-ttu-id="d8801-105">Отменяются или запрещаются разрешения на все базовые объекты (например, таблицы), на хранимые процедуры предоставляются разрешения EXECUTE.</span><span class="sxs-lookup"><span data-stu-id="d8801-105">You revoke or deny all permissions to underlying objects, such as tables, and grant EXECUTE permissions on stored procedures.</span></span> <span data-ttu-id="d8801-106">Тем самым создается эффективный периметр безопасности вокруг данных и объектов базы данных.</span><span class="sxs-lookup"><span data-stu-id="d8801-106">This effectively creates a security perimeter around your data and database objects.</span></span>  
  
## <a name="stored-procedure-benefits"></a><span data-ttu-id="d8801-107">Преимущества хранимых процедур</span><span class="sxs-lookup"><span data-stu-id="d8801-107">Stored Procedure Benefits</span></span>  

 <span data-ttu-id="d8801-108">Хранимые процедуры обладают следующими преимуществами.</span><span class="sxs-lookup"><span data-stu-id="d8801-108">Stored procedures have the following benefits:</span></span>  
  
- <span data-ttu-id="d8801-109">Логику данных и бизнес-правила можно инкапсулировать, чтобы пользователи получали доступ к данным и объектам только тем путем, который предусмотрен разработчиками и администраторами баз данных.</span><span class="sxs-lookup"><span data-stu-id="d8801-109">Data logic and business rules can be encapsulated so that users can access data and objects only in ways that developers and database administrators intend.</span></span>  
  
- <span data-ttu-id="d8801-110">Чтобы исключить возможность атак путем внедрения кода SQL, можно использовать параметризованные хранимые процедуры, проверяющие ввод пользователя.</span><span class="sxs-lookup"><span data-stu-id="d8801-110">Parameterized stored procedures that validate all user input can be used to thwart SQL injection attacks.</span></span> <span data-ttu-id="d8801-111">При использовании динамического SQL обязательно параметризуйте команды и ни в коем случае не включайте значения параметров непосредственно в строку запроса.</span><span class="sxs-lookup"><span data-stu-id="d8801-111">If you use dynamic SQL, be sure to parameterize your commands, and never include parameter values directly into a query string.</span></span>  
  
- <span data-ttu-id="d8801-112">Можно запретить нерегламентированные запросы и изменение данных.</span><span class="sxs-lookup"><span data-stu-id="d8801-112">Ad hoc queries and data modifications can be disallowed.</span></span> <span data-ttu-id="d8801-113">Таким образом пользователи не смогут злонамеренно или случайно уничтожить данные или выполнить запросы, снижающие производительность сервера в сети.</span><span class="sxs-lookup"><span data-stu-id="d8801-113">This prevents users from maliciously or inadvertently destroying data or executing queries that impair performance on the server or the network.</span></span>  
  
- <span data-ttu-id="d8801-114">В коде процедур можно обрабатывать ошибки, не передавая их в клиентские приложения.</span><span class="sxs-lookup"><span data-stu-id="d8801-114">Errors can be handled in procedure code without being passed directly to client applications.</span></span> <span data-ttu-id="d8801-115">Таким образом предотвращается возвращение сообщений об ошибках, которые могут использоваться в атаках зондированием.</span><span class="sxs-lookup"><span data-stu-id="d8801-115">This prevents error messages from being returned that could aid in a probing attack.</span></span> <span data-ttu-id="d8801-116">Регистрируйте ошибки и обрабатывайте их на сервере.</span><span class="sxs-lookup"><span data-stu-id="d8801-116">Log errors and handle them on the server.</span></span>  
  
- <span data-ttu-id="d8801-117">К написанным однажды хранимым процедурам могут обращаться многие приложения.</span><span class="sxs-lookup"><span data-stu-id="d8801-117">Stored procedures can be written once, and accessed by many applications.</span></span>  
  
- <span data-ttu-id="d8801-118">Клиентским приложениям не требуются сведения о базовых структурах данных.</span><span class="sxs-lookup"><span data-stu-id="d8801-118">Client applications do not need to know anything about the underlying data structures.</span></span> <span data-ttu-id="d8801-119">Код хранимых процедур можно изменять, не внося изменения в клиентские приложения, если это не влияет на списки параметров и возвращаемые типы данных.</span><span class="sxs-lookup"><span data-stu-id="d8801-119">Stored procedure code can be changed without requiring changes in client applications as long as the changes do not affect parameter lists or returned data types.</span></span>  
  
- <span data-ttu-id="d8801-120">Хранимые процедуры снижают нагрузку на сетевой трафик, объединяя несколько операций в одном вызове процедуры.</span><span class="sxs-lookup"><span data-stu-id="d8801-120">Stored procedures can reduce network traffic by combining multiple operations into one procedure call.</span></span>  
  
## <a name="stored-procedure-execution"></a><span data-ttu-id="d8801-121">Выполнение хранимых процедур</span><span class="sxs-lookup"><span data-stu-id="d8801-121">Stored Procedure Execution</span></span>  

 <span data-ttu-id="d8801-122">Хранимые процедуры предоставляют доступ к данным с помощью цепочек владения таким образом, чтобы пользователям не требовались явные разрешения для работы с объектами базы данных.</span><span class="sxs-lookup"><span data-stu-id="d8801-122">Stored procedures take advantage of ownership chaining to provide access to data so that users do not need to have explicit permission to access database objects.</span></span> <span data-ttu-id="d8801-123">Цепочка владения возникает, когда объекты, последовательно получающие доступ друг к другу, принадлежат одному пользователю.</span><span class="sxs-lookup"><span data-stu-id="d8801-123">An ownership chain exists when objects that access each other sequentially are owned by the same user.</span></span> <span data-ttu-id="d8801-124">Например, хранимая процедура может вызывать другие хранимые процедуры или обращаться к нескольким таблицам.</span><span class="sxs-lookup"><span data-stu-id="d8801-124">For example, a stored procedure can call other stored procedures, or a stored procedure can access multiple tables.</span></span> <span data-ttu-id="d8801-125">Если все объекты в цепочке владения принадлежат одному владельцу, то SQL Server проверяет только разрешение EXECUTE вызывающего, но не разрешения вызывающего на другие объекты.</span><span class="sxs-lookup"><span data-stu-id="d8801-125">If all objects in the chain of execution have the same owner, then SQL Server only checks the EXECUTE permission for the caller, not the caller's permissions on other objects.</span></span> <span data-ttu-id="d8801-126">Поэтому нужно предоставить только разрешения EXECUTE на хранимые процедуры и отменить или запретить все разрешения на базовые таблицы.</span><span class="sxs-lookup"><span data-stu-id="d8801-126">Therefore you need to grant only EXECUTE permissions on stored procedures; you can revoke or deny all permissions on the underlying tables.</span></span>  
  
## <a name="best-practices"></a><span data-ttu-id="d8801-127">Рекомендации</span><span class="sxs-lookup"><span data-stu-id="d8801-127">Best Practices</span></span>  

 <span data-ttu-id="d8801-128">Простого написания хранимой процедуры недостаточно для надежной защиты приложения.</span><span class="sxs-lookup"><span data-stu-id="d8801-128">Simply writing stored procedures isn't enough to adequately secure your application.</span></span> <span data-ttu-id="d8801-129">Необходимо учитывать следующие потенциальные уязвимости защиты.</span><span class="sxs-lookup"><span data-stu-id="d8801-129">You should also consider the following potential security holes.</span></span>  
  
- <span data-ttu-id="d8801-130">Предоставьте разрешения EXECUTE на хранимые процедуры нужным ролям базы данных для доступа к данным.</span><span class="sxs-lookup"><span data-stu-id="d8801-130">Grant EXECUTE permissions on the stored procedures for database roles you want to be able to access the data.</span></span>  
  
- <span data-ttu-id="d8801-131">Отмените или запретите все разрешения на базовые таблицы для всех ролей и пользователей базы данных, в том числе роль `public`.</span><span class="sxs-lookup"><span data-stu-id="d8801-131">Revoke or deny all permissions to the underlying tables for all roles and users in the database, including the `public` role.</span></span> <span data-ttu-id="d8801-132">Все пользователи наследуют разрешения из роли public.</span><span class="sxs-lookup"><span data-stu-id="d8801-132">All users inherit permissions from public.</span></span> <span data-ttu-id="d8801-133">Поэтому запрет разрешений для роли `public` означает, что доступ будут иметь только владельцы объектов и члены роли `sysadmin`, все остальные пользователи не смогут наследовать разрешения, будучи членами других ролей.</span><span class="sxs-lookup"><span data-stu-id="d8801-133">Therefore denying permissions to `public` means that only owners and `sysadmin` members have access; all other users will be unable to inherit permissions from membership in other roles.</span></span>  
  
- <span data-ttu-id="d8801-134">Не добавляйте пользователей или роли к ролям `sysadmin` и `db_owner`.</span><span class="sxs-lookup"><span data-stu-id="d8801-134">Do not add users or roles to the `sysadmin` or `db_owner` roles.</span></span> <span data-ttu-id="d8801-135">Системные администраторы и владельцы базы данных имеют доступ ко всем объектам базы данных.</span><span class="sxs-lookup"><span data-stu-id="d8801-135">System administrators and database owners can access all database objects.</span></span>  
  
- <span data-ttu-id="d8801-136">Отключите учетную запись `guest`.</span><span class="sxs-lookup"><span data-stu-id="d8801-136">Disable the `guest` account.</span></span> <span data-ttu-id="d8801-137">Это будет препятствовать соединению анонимных пользователей с базой данных.</span><span class="sxs-lookup"><span data-stu-id="d8801-137">This will prevent anonymous users from connecting to the database.</span></span> <span data-ttu-id="d8801-138">В новых базах данных учетная запись guest отключена по умолчанию.</span><span class="sxs-lookup"><span data-stu-id="d8801-138">The guest account is disabled by default in new databases.</span></span>  
  
- <span data-ttu-id="d8801-139">Реализуйте обработку ошибок и ошибочных записей журнала.</span><span class="sxs-lookup"><span data-stu-id="d8801-139">Implement error handling and log errors.</span></span>  
  
- <span data-ttu-id="d8801-140">Создайте параметризованные хранимые процедуры, которые будут проверять ввод пользователей.</span><span class="sxs-lookup"><span data-stu-id="d8801-140">Create parameterized stored procedures that validate all user input.</span></span> <span data-ttu-id="d8801-141">Считайте все входные данные пользователей не соответствующими требованиям безопасности.</span><span class="sxs-lookup"><span data-stu-id="d8801-141">Treat all user input as untrusted.</span></span>  
  
- <span data-ttu-id="d8801-142">Избегайте использования динамического SQL за исключением абсолютно необходимых случаев.</span><span class="sxs-lookup"><span data-stu-id="d8801-142">Avoid dynamic SQL unless absolutely necessary.</span></span> <span data-ttu-id="d8801-143">Используйте функцию Transact-SQL QUOTENAME() для разделения строковых значений и избегайте любого вхождения разделителей в строке ввода.</span><span class="sxs-lookup"><span data-stu-id="d8801-143">Use the Transact-SQL QUOTENAME() function to delimit a string value and escape any occurrence of the delimiter in the input string.</span></span>
