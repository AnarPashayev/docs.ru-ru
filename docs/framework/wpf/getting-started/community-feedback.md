# <a name="wpf-community-feedback"></a>Отзывы сообщества WPF

Корпорация Майкрософт предоставляет разнообразные ресурсы сообщества для изучения, обсуждения и предоставления отзывов о Windows Presentation Foundation (WPF). Эти ресурсы включают форумы и сайт [сообщества разработчиков Visual Studio](https://developercommunity.visualstudio.com/) . Каждый ресурс сообщества предлагает различный набор услуг. Эти преимущества описаны здесь, как и набор рекомендаций по использованию каждого из них для обеспечения наилучшего ответа от сообщества в крупных и корпораций Майкрософт.

> [!NOTE]
> Не используйте раздел Обратная связь, расположенный в нижней части каждой страницы, чтобы отправить отзыв о продукте. Эти ссылки служат только для отзывов о документации.

## <a name="forums"></a>Форумы

[Форум WPF](https://social.msdn.microsoft.com/Forums/vstudio/home?forum=wpf) — это основной ресурс сообщества для обсуждения и устранения проблем. Форумы облегчают обсуждение и разрешение проблем, предлагая набор функциональных возможностей, включающий следующие.

- Поиск.
- Слежение за обсуждением.
- Расширенное форматирование текста и кода.
- Интеграция Visual Studio.
- Привлечение членов MVP и участие в сообществах.
- Наблюдение за тем, чтобы на сообщения были даны ответы в кратчайшее время.

Кроме того, вы можете задать вопросы в сообществе, посвященном WPF, [Stack overflow](https://stackoverflow.com/questions/tagged/wpf).

### <a name="forum-best-practices"></a>Рекомендации на форуме

Следующие рекомендации помогут устранить проблемы, опубликованные на форуме WPF, в кратчайшие моменты времени. Эти рекомендации применимы для всех форумов.

#### <a name="search-existing-posts"></a>Поиск существующих записей

Некоторые вопросы возникают достаточно часто, поэтому другие пользователи могли столкнуться с ними до вас. Следовательно, вы можете быстро решить свою проблему или внести свой вклад в существующее обсуждение.

#### <a name="use-meaningful-titles"></a>Использование содержательных заголовков

Краткие и осмысленные названия улучшают возможность обнаружения ваших сообщений. Они также облегчают другим участникам сообщества форумов по WPF возможность определить, могут ли они решить вашу проблему.

#### <a name="include-appropriate-content"></a>Включить соответствующее содержимое

Опишите проблемы и предпринимали попыток работы с ней. По возможности включите фрагменты кода или простейший образец, который демонстрирует проблему. Все эти сведения увеличивают вероятность быстрого ответа на вопрос.

## <a name="visual-studio-developer-community"></a>Сообщество разработчиков Visual Studio

Иногда вопросы могут быть трудно разрешимыми или неразрешимыми. Такие ситуации возникают из-за ошибок в технологии, трудностей применения технологии в конкретных скриптах или отсутствия поддержки для конкретных скриптов. Эта информация важна для корпорации Майкрософт и может быть предоставлена на веб-сайте [сообщества разработчиков Visual Studio](https://developercommunity.visualstudio.com/) .

Элементы, размещенные в центре отзывов по продуктам WPF, направляются в базу данных внутренней ошибки команды WPF. Таким образом, это самый надежный способ получить отзыв о владельце функции WPF. Кроме того, вы можете проверить и отслеживание предложений и ошибок, а также проголосовать за них, что помогает группе разработчиков WPF расставить приоритеты проблем.

### <a name="developer-community-best-practices"></a>Рекомендации сообщества разработчиков

При размещении в сообществе разработчиков Visual Studio Поиск по существующим сообщениям, предоставление осмысленного названия и соответствующего содержимого являются важными рекомендациями, так же, как и для публикации на форуме WPF. Ниже приведены дополнительные рекомендации, которые также следует применять.

#### <a name="search-existing-posts"></a>Поиск существующих записей

Некоторые вопросы возникают достаточно часто, поэтому другие пользователи могли столкнуться с ними до вас. Следовательно, можно быстро решить проблему или добавить свои входные данные в существующую проблему.

#### <a name="use-meaningful-titles"></a>Использование содержательных заголовков

Краткие, понятные названия увеличивают вероятность того, что ваша ошибка будет направлена в наиболее подходящую группу разработчиков WPF в кратчайший промежуток времени. Это особенно важно для такой технологии, как WPF, которая содержит множество взаимосвязанных функций.

#### <a name="describe-how-to-reproduce-your-bug"></a>Описание способов воспроизведения ошибки

При отправке сообщения об ошибке важно включить следующие сведения.

- Предоставить четкое описание ошибки.
- Использовать фрагменты программного кода для иллюстрации описания ошибки.
- Предоставить последовательность шагов для воспроизведения ошибки.
- Включить наименьший образец кода, воспроизводящий ошибку.
- Указать, постоянно ли возникает ошибка.
- Включить информацию о соответствующем исключении.

 Если ошибка связана с установкой или настройкой, приложите соответствующие журналы установки и снимки (файлы с префиксом dd_, расположенные в папке %temp%).

 Для ошибок компиляции или построения приложите журналы построения. Система MSBuild может быть настроена для поддержки ведения журнала с различными уровнями детализации с помощью параметра/v: из командной строки или путем настройки соответствующего уровня из интегрированной среды разработки (IDE), например Visual Studio.

#### <a name="provide-environment-information"></a>Укажите сведения о среде

Информация о среде полезна для обеспечения контекста сообщения. В частности, следует упомянуть о платформе операционной системы, семействе процессоров и архитектуре, например "Windows 10 версии 1709, Intel (R) Xeon (R), x64".

Если отсылаемый вопрос связан с отрисовкой, следует по возможности включить сведения о видеокарте и драйвере. Эта информация важна, поскольку WPF является платформой представления.

#### <a name="provide-solution-or-project-information"></a>Предоставление сведений о решении или проекте

Ошибки могут относиться к средствам, используемым при разработке и построении приложений и типов приложений. Следовательно, может быть полезно указать следующее.

- Тип приложения, для которого выполняется построение.
  - Приложение (*exe*) или библиотека ( *. dll*)
  - Приложение браузера XAML (XAML) (XBAP)
  - Свободное приложение XAML
  - Автономные установленные приложения
  - Автономные приложения, развернутые с помощью ClickOnce
- Средство разработки.
  - MSBuild
  - Конструктор графики выражений
  - Интерактивный конструктор выражений
  - Visual Studio
- Конфигурация решения.
  - Решение
  - Один проект
  - Решение с несколькими зависимыми проектами
- Имеет ли приложение ресурсы для конкретного языка или ресурсы, нейтральные для языка. Например, указано ли свойство проекта `UICulture` или локализуемые метаданные для типов `Application`, `Page` и `Resource`?
- Использован ли параметр нейтрального языка в файле AssemblyInfo.cs или AssemblyInfo.vb.

#### <a name="provide-scenario-and-impact-information"></a>Предоставление сведений о сценарии и влиянии

Предоставьте сведения о сценарии, который проявляется манифестом ошибки и ее влиянием. Эта информация очень важна для команды разработчиков WPF, когда она определяет, когда и как следует исправить проблему, а также можно ли использовать приемлемое решение.

Ситуации сбоя и потери данных обычно имеют высокое влияние, и таким образом, для них проще всего назначить приоритет при решении проблемы. Некоторые ошибки отображаются только в нечасто используемых скриптах, которые, однако, в некоторых случаях могут оказаться основными. Предоставление контекста для сценария и влияния помогает группе разработчиков WPF принять правильное решение.
