# <a name="build-resilient-services-ready-for-the-cloud-embrace-transient-failures-in-the-cloud"></a><span data-ttu-id="1d15e-105">Создание устойчивых служб, готовых к использованию в облаке: адаптация к временным сбоям в облаке</span><span class="sxs-lookup"><span data-stu-id="1d15e-105">Build resilient services ready for the cloud: Embrace transient failures in the cloud</span></span>

<span data-ttu-id="1d15e-106">Устойчивость — это возможность восстановления после сбоев и продолжение работы.</span><span class="sxs-lookup"><span data-stu-id="1d15e-106">Resiliency is the ability to recover from failures and continue to function.</span></span> <span data-ttu-id="1d15e-107">Устойчивость не сводится к предотвращению сбоев, но принимает тот факт, что произойдет сбой, и затем отреагировать на них таким образом, чтобы избежать простоя или потери данных.</span><span class="sxs-lookup"><span data-stu-id="1d15e-107">Resiliency is not about avoiding failures, but accepting the fact that failures will occur, and then responding to them in a way that avoids downtime or data loss.</span></span> <span data-ttu-id="1d15e-108">Цель устойчивости — вернуть приложение в полностью функционирующее состояние после сбоя.</span><span class="sxs-lookup"><span data-stu-id="1d15e-108">The goal of resiliency is to return the application to a fully functioning state after a failure.</span></span>

<span data-ttu-id="1d15e-109">Приложение готово к работе в облаке, когда, как минимум, оно реализует программную модель устойчивости, а не модель на основе оборудования.</span><span class="sxs-lookup"><span data-stu-id="1d15e-109">Your application is ready for the cloud when, at a minimum, it implements a software-based model of resiliency, rather than a hardware-based model.</span></span> <span data-ttu-id="1d15e-110">Облачное приложение должно принять в себя частичные сбои, которые, безусловно, будут выполняться.</span><span class="sxs-lookup"><span data-stu-id="1d15e-110">Your cloud application must embrace the partial failures that will certainly occur.</span></span> <span data-ttu-id="1d15e-111">Создайте или частично выполните рефакторинг приложения, чтобы добиться устойчивости с ожидаемыми частичными сбоями.</span><span class="sxs-lookup"><span data-stu-id="1d15e-111">Design or partially refactor your application to achieve resiliency with expected partial failures.</span></span> <span data-ttu-id="1d15e-112">Она должна быть разработана для работы с частичными сбоями, такими как временные сбои сети, узлы или сбои виртуальных машин в облаке.</span><span class="sxs-lookup"><span data-stu-id="1d15e-112">It should be designed to cope with partial failures, like transient network outages and nodes, or VMs crashing in the cloud.</span></span> <span data-ttu-id="1d15e-113">Даже контейнеры, перемещаемые в другой узел в кластере Orchestrator, могут вызвать периодические кратковременные сбои в приложении.</span><span class="sxs-lookup"><span data-stu-id="1d15e-113">Even containers being moved to a different node within an orchestrator cluster can cause intermittent short failures within the application.</span></span>

## <a name="handling-partial-failure"></a><span data-ttu-id="1d15e-114">Обработка частичного сбоя</span><span class="sxs-lookup"><span data-stu-id="1d15e-114">Handling partial failure</span></span>

<span data-ttu-id="1d15e-115">В облачном приложении существует риск частичного сбоя.</span><span class="sxs-lookup"><span data-stu-id="1d15e-115">In a cloud-based application, there's an ever-present risk of partial failure.</span></span> <span data-ttu-id="1d15e-116">Например, один экземпляр веб-сайта или контейнер может завершиться ошибкой или в течение короткого времени может быть недоступен или не отвечает.</span><span class="sxs-lookup"><span data-stu-id="1d15e-116">For instance, a single website instance or a container might fail, or it might be unavailable or unresponsive for a short time.</span></span> <span data-ttu-id="1d15e-117">Или может произойти сбой одной виртуальной машины или сервера.</span><span class="sxs-lookup"><span data-stu-id="1d15e-117">Or, a single VM or server might crash.</span></span>

<span data-ttu-id="1d15e-118">Поскольку клиенты и службы являются отдельными процессами, служба не может своевременно отвечать на запросы клиента.</span><span class="sxs-lookup"><span data-stu-id="1d15e-118">Because clients and services are separate processes, a service might not be able to respond in a timely manner to a client's request.</span></span> <span data-ttu-id="1d15e-119">Служба может быть перегружена и медленнее реагировать на запросы, или она может быть недоступна в течение короткого промежутка времени из-за проблем с сетью.</span><span class="sxs-lookup"><span data-stu-id="1d15e-119">The service might be overloaded and respond slowly to requests, or it might not be accessible for a short time because of network issues.</span></span>

<span data-ttu-id="1d15e-120">Например, рассмотрим монолитное приложение .NET, которое обращается к базе данных в базе данных SQL Azure.</span><span class="sxs-lookup"><span data-stu-id="1d15e-120">For example, consider a monolithic .NET application that accesses a database in Azure SQL Database.</span></span> <span data-ttu-id="1d15e-121">Если база данных SQL Azure или любая другая служба стороннего производителя не отвечает на короткое время (база данных SQL Azure может быть перемещена на другой узел или сервер, не отвечают на несколько секунд), то при попытке пользователя выполнить какое-либо действие приложение может завершиться сбоем и попытаться в то же время исключение.</span><span class="sxs-lookup"><span data-stu-id="1d15e-121">If the Azure SQL database or any other third-party service is unresponsive for a brief time (an Azure SQL database might be moved to a different node or server, and be unresponsive for a few seconds), when the user tries to do any action, the application might crash and show an exception at the same moment.</span></span>

<span data-ttu-id="1d15e-122">Подобный сценарий может возникать в приложении, использующем службы HTTP.</span><span class="sxs-lookup"><span data-stu-id="1d15e-122">A similar scenario might occur in an app that consumes HTTP services.</span></span> <span data-ttu-id="1d15e-123">Сеть или сама служба могут быть недоступны в облаке во время короткого временного сбоя.</span><span class="sxs-lookup"><span data-stu-id="1d15e-123">The network or the service itself might not be available in the cloud during a short, transient failure.</span></span>

<span data-ttu-id="1d15e-124">Устойчивое приложение, как показано на рисунке 4-9, должно реализовать такие приемы, как "повторные попытки с экспоненциальной задержкой", чтобы дать приложению возможность управлять временными сбоями в ресурсах.</span><span class="sxs-lookup"><span data-stu-id="1d15e-124">A resilient application like the one shown in Figure 4-9 should implement techniques like "retries with exponential backoff" to give the application an opportunity to handle transient failures in resources.</span></span> <span data-ttu-id="1d15e-125">В приложениях также следует использовать "автоматическое разбиение".</span><span class="sxs-lookup"><span data-stu-id="1d15e-125">You also should use "circuit breakers" in your applications.</span></span> <span data-ttu-id="1d15e-126">Автоматическое выключение прекращает попытки приложения получить доступ к ресурсу, когда на самом деле происходит долгосрочный сбой.</span><span class="sxs-lookup"><span data-stu-id="1d15e-126">A circuit breaker stops an application from trying to access a resource when it's actually a long-term failure.</span></span> <span data-ttu-id="1d15e-127">С помощью автоматического выключения приложение позволяет избежать заставляющегоия отказа в обслуживании.</span><span class="sxs-lookup"><span data-stu-id="1d15e-127">By using a circuit breaker, the application avoids provoking a denial of service to itself.</span></span>
