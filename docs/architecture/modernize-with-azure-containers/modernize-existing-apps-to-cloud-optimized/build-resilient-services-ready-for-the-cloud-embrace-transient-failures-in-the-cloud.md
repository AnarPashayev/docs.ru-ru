# <a name="build-resilient-services-ready-for-the-cloud-embrace-transient-failures-in-the-cloud"></a><span data-ttu-id="55452-105">Создание устойчивых служб, готовых к использованию в облаке: адаптация к временным сбоям в облаке</span><span class="sxs-lookup"><span data-stu-id="55452-105">Build resilient services ready for the cloud: Embrace transient failures in the cloud</span></span>

<span data-ttu-id="55452-106">Устойчивость — это возможность восстановления после сбоев и продолжение работы.</span><span class="sxs-lookup"><span data-stu-id="55452-106">Resiliency is the ability to recover from failures and continue to function.</span></span> <span data-ttu-id="55452-107">Исходить следует из того, что избежать сбоев невозможно и на них нужно реагировать так, чтобы избежать простоев или потери данных.</span><span class="sxs-lookup"><span data-stu-id="55452-107">Resiliency is not about avoiding failures, but accepting the fact that failures will occur, and then responding to them in a way that avoids downtime or data loss.</span></span> <span data-ttu-id="55452-108">Цель устойчивости — вернуть приложение в полностью функционирующее состояние после сбоя.</span><span class="sxs-lookup"><span data-stu-id="55452-108">The goal of resiliency is to return the application to a fully functioning state after a failure.</span></span>

<span data-ttu-id="55452-109">Приложение готово к работе в облаке тогда, когда оно по крайней мере реализует программную модель устойчивости, а не аппаратную.</span><span class="sxs-lookup"><span data-stu-id="55452-109">Your application is ready for the cloud when, at a minimum, it implements a software-based model of resiliency, rather than a hardware-based model.</span></span> <span data-ttu-id="55452-110">Для облачного приложения необходимо предусмотреть частичные сбои, которые, безусловно, будут происходить.</span><span class="sxs-lookup"><span data-stu-id="55452-110">Your cloud application must embrace the partial failures that will certainly occur.</span></span> <span data-ttu-id="55452-111">Создайте приложение или выполните его частичный рефакторинг, чтобы добиться устойчивости при ожидаемых частичных сбоях.</span><span class="sxs-lookup"><span data-stu-id="55452-111">Design or partially refactor your application to achieve resiliency with expected partial failures.</span></span> <span data-ttu-id="55452-112">Это значит, что оно должно справляться с такими сбоями, как временные отключения сети либо аварийные сбои узлов или виртуальных машин в облаке.</span><span class="sxs-lookup"><span data-stu-id="55452-112">It should be designed to cope with partial failures, like transient network outages and nodes, or VMs crashing in the cloud.</span></span> <span data-ttu-id="55452-113">Даже перенос контейнеров в другой узел кластера оркестратора может приводить к кратковременным сбоям в приложении.</span><span class="sxs-lookup"><span data-stu-id="55452-113">Even containers being moved to a different node within an orchestrator cluster can cause intermittent short failures within the application.</span></span>

## <a name="handling-partial-failure"></a><span data-ttu-id="55452-114">Обработка частичного сбоя</span><span class="sxs-lookup"><span data-stu-id="55452-114">Handling partial failure</span></span>

<span data-ttu-id="55452-115">В облачном приложении всегда есть риск частичного сбоя.</span><span class="sxs-lookup"><span data-stu-id="55452-115">In a cloud-based application, there's an ever-present risk of partial failure.</span></span> <span data-ttu-id="55452-116">Например, один экземпляр веб-сайта или контейнер может быть недоступен, не отвечать в течение короткого времени или может произойти его сбой.</span><span class="sxs-lookup"><span data-stu-id="55452-116">For instance, a single website instance or a container might fail, or it might be unavailable or unresponsive for a short time.</span></span> <span data-ttu-id="55452-117">Может также произойти аварийный сбой одной виртуальной машины или сервера.</span><span class="sxs-lookup"><span data-stu-id="55452-117">Or, a single VM or server might crash.</span></span>

<span data-ttu-id="55452-118">Так как клиенты и службы работают отдельными процессами, служба может не ответить вовремя на запрос клиента.</span><span class="sxs-lookup"><span data-stu-id="55452-118">Because clients and services are separate processes, a service might not be able to respond in a timely manner to a client's request.</span></span> <span data-ttu-id="55452-119">Служба может быть перегружена, медленно отвечать на запросы или быть недоступна некоторое время из-за проблем с сетью.</span><span class="sxs-lookup"><span data-stu-id="55452-119">The service might be overloaded and respond slowly to requests, or it might not be accessible for a short time because of network issues.</span></span>

<span data-ttu-id="55452-120">Например, рассмотрим монолитное приложение .NET, которое обращается к базе в Базе данных SQL Azure.</span><span class="sxs-lookup"><span data-stu-id="55452-120">For example, consider a monolithic .NET application that accesses a database in Azure SQL Database.</span></span> <span data-ttu-id="55452-121">Если База данных SQL Azure или любая сторонняя служба не отвечает в течение небольшого периода времени (если база перемещена на другой узел или сервер, она не будет отвечать несколько секунд), то при попытке пользователя выполнить какое-либо действие может одновременно произойти аварийное завершение работы приложения и выдача исключения.</span><span class="sxs-lookup"><span data-stu-id="55452-121">If the Azure SQL database or any other third-party service is unresponsive for a brief time (an Azure SQL database might be moved to a different node or server, and be unresponsive for a few seconds), when the user tries to do any action, the application might crash and show an exception at the same moment.</span></span>

<span data-ttu-id="55452-122">Подобный сценарий может возникать в приложении, использующем HTTP-службы.</span><span class="sxs-lookup"><span data-stu-id="55452-122">A similar scenario might occur in an app that consumes HTTP services.</span></span> <span data-ttu-id="55452-123">Сеть или сама служба могут быть недоступны в облаке во время короткого сбоя.</span><span class="sxs-lookup"><span data-stu-id="55452-123">The network or the service itself might not be available in the cloud during a short, transient failure.</span></span>

<span data-ttu-id="55452-124">Чтобы приложение, например показанное на рисунке 4-9, было устойчивым, в нем следует реализовать такие методы, как "повторные попытки с экспоненциальной задержкой", чтобы дать возможность справляться со временными сбоями ресурсов.</span><span class="sxs-lookup"><span data-stu-id="55452-124">A resilient application like the one shown in Figure 4-9 should implement techniques like "retries with exponential backoff" to give the application an opportunity to handle transient failures in resources.</span></span> <span data-ttu-id="55452-125">В приложениях также следует использовать автоматическое выключение.</span><span class="sxs-lookup"><span data-stu-id="55452-125">You also should use "circuit breakers" in your applications.</span></span> <span data-ttu-id="55452-126">Это значит, что приложение будет прекращать попытки получить доступ к ресурсу, когда на самом деле происходит долгосрочный сбой.</span><span class="sxs-lookup"><span data-stu-id="55452-126">A circuit breaker stops an application from trying to access a resource when it's actually a long-term failure.</span></span> <span data-ttu-id="55452-127">Автоматическое выключение приложения позволяет не провоцировать отказ в обслуживании.</span><span class="sxs-lookup"><span data-stu-id="55452-127">By using a circuit breaker, the application avoids provoking a denial of service to itself.</span></span>

<span data-ttu-id="55452-131">Эти методы можно использовать как с HTTP-ресурсами, так и с ресурсами базы данных.</span><span class="sxs-lookup"><span data-stu-id="55452-131">You can use these techniques both in HTTP resources and in database resources.</span></span> <span data-ttu-id="55452-132">На рис. 4-9 показано приложение с трехуровневой архитектурой, поэтому методы необходимо применять на уровне служб (HTTP) и на уровне данных (TCP).</span><span class="sxs-lookup"><span data-stu-id="55452-132">In Figure 4-9, the application is based on a 3-tier architecture, so you need these techniques at the services level (HTTP) and at the data tier level (TCP).</span></span> <span data-ttu-id="55452-133">В монолитном приложении, которое использует только один уровень в дополнение к базе данных (без дополнительных служб или микрослужб), может быть достаточно обработки временных сбоев на уровне подключения к базе данных.</span><span class="sxs-lookup"><span data-stu-id="55452-133">In a monolithic application that uses only a single app tier in addition to the database (no additional services or microservices), handling transient failures at the database connection level might be enough.</span></span> <span data-ttu-id="55452-134">В этом сценарии требуется только определенная настройка подключения к базе данных.</span><span class="sxs-lookup"><span data-stu-id="55452-134">In that scenario, just a particular configuration of the database connection is required.</span></span>

<span data-ttu-id="55452-135">Это может не вызывать затруднений при реализации отказоустойчивых подключений с доступом к базе данных, в зависимости от используемой версии .NET (например, для [Entity Framework 6 или более поздней версии](/ef/ef6/fundamentals/connection-resiliency/retry-logic)).</span><span class="sxs-lookup"><span data-stu-id="55452-135">When implementing resilient communications that access the database, depending on the version of .NET you are using, it can be straightforward (for example, [with Entity Framework 6 or later](/ef/ef6/fundamentals/connection-resiliency/retry-logic).</span></span> <span data-ttu-id="55452-136">Необходимо всего лишь настроить подключение к базе данных.</span><span class="sxs-lookup"><span data-stu-id="55452-136">It's just a matter of configuring the database connection).</span></span> <span data-ttu-id="55452-137">Но может также потребоваться использовать дополнительные библиотеки, такие как [Блок приложения для обработки временной ошибки](https://docs.microsoft.com/previous-versions/msp-n-p/hh680934(v=pandp.50)) (для .NET предыдущих версий), или даже внедрить собственную библиотеку.</span><span class="sxs-lookup"><span data-stu-id="55452-137">Or, you might need to use additional libraries like the [Transient Fault Handling Application Block](https://docs.microsoft.com/previous-versions/msp-n-p/hh680934(v=pandp.50)) (for earlier versions of .NET), or even implement your own library.</span></span>

<span data-ttu-id="55452-138">При реализации повторных попыток HTTP и автоматического отключения следует использовать библиотеку [Polly](https://github.com/App-vNext/Polly), предназначенную для .NET Framework 4.0, 4.5 и .NET Standard 1.1 с поддержкой .NET Core.</span><span class="sxs-lookup"><span data-stu-id="55452-138">When implementing HTTP retries and circuit breakers, the recommendation for .NET is to use the [Polly](https://github.com/App-vNext/Polly) library, which targets .NET Framework 4.0, .NET Framework 4.5, and .NET Standard 1.1, which includes .NET Core support.</span></span>
