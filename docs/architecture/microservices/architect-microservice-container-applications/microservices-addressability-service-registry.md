# <a name="microservices-addressability-and-the-service-registry"></a><span data-ttu-id="76aae-103">Возможность адресации микрослужб и реестр служб</span><span class="sxs-lookup"><span data-stu-id="76aae-103">Microservices addressability and the service registry</span></span>

<span data-ttu-id="76aae-104">Каждая микрослужба имеет уникальное имя (URL-адрес), по которому разрешается ее расположение.</span><span class="sxs-lookup"><span data-stu-id="76aae-104">Each microservice has a unique name (URL) that's used to resolve its location.</span></span> <span data-ttu-id="76aae-105">Возможность адресации микрослужбы должна существовать независимо от того, где она находится.</span><span class="sxs-lookup"><span data-stu-id="76aae-105">Your microservice needs to be addressable wherever it's running.</span></span> <span data-ttu-id="76aae-106">Если вы не можете легко определить, на каком компьютере выполняется та или иная микрослужба, очень скоро могут возникнуть проблемы.</span><span class="sxs-lookup"><span data-stu-id="76aae-106">If you have to think about which computer is running a particular microservice, things can go bad quickly.</span></span> <span data-ttu-id="76aae-107">Микрослужба должна иметь уникальное имя, чтобы ее текущее расположение можно было установить аналогично тому, как служба DNS сопоставляет URL-адрес с определенным компьютером.</span><span class="sxs-lookup"><span data-stu-id="76aae-107">In the same way that DNS resolves a URL to a particular computer, your microservice needs to have a unique name so that its current location is discoverable.</span></span> <span data-ttu-id="76aae-108">Микрослужбам требуются адресуемые имена, которые делают их независимыми от базовой инфраструктуры.</span><span class="sxs-lookup"><span data-stu-id="76aae-108">Microservices need addressable names that make them independent from the infrastructure that they're running on.</span></span> <span data-ttu-id="76aae-109">Это означает наличие связи между развертыванием службы и ее обнаружением, то есть необходим [реестр служб](https://microservices.io/patterns/service-registry.html).</span><span class="sxs-lookup"><span data-stu-id="76aae-109">This implies that there's an interaction between how your service is deployed and how it's discovered, because there needs to be a [service registry](https://microservices.io/patterns/service-registry.html).</span></span> <span data-ttu-id="76aae-110">Аналогичным образом, когда происходит сбой компьютера, служба реестра должна сообщать, где теперь выполняется служба.</span><span class="sxs-lookup"><span data-stu-id="76aae-110">In the same vein, when a computer fails, the registry service must be able to indicate where the service is now running.</span></span>

<span data-ttu-id="76aae-111">[Шаблон реестра служб](https://microservices.io/patterns/service-registry.html) является ключевым элементом обнаружения служб.</span><span class="sxs-lookup"><span data-stu-id="76aae-111">The [service registry pattern](https://microservices.io/patterns/service-registry.html) is a key part of service discovery.</span></span> <span data-ttu-id="76aae-112">Он представляет собой базу данных, содержащую сетевые расположения экземпляров служб.</span><span class="sxs-lookup"><span data-stu-id="76aae-112">The registry is a database containing the network locations of service instances.</span></span> <span data-ttu-id="76aae-113">Реестр служб должен обладать высокой доступностью и быть актуальным.</span><span class="sxs-lookup"><span data-stu-id="76aae-113">A service registry needs to be highly available and up-to-date.</span></span> <span data-ttu-id="76aae-114">Клиенты могут кэшировать сетевые расположения, полученные из реестра служб.</span><span class="sxs-lookup"><span data-stu-id="76aae-114">Clients could cache network locations obtained from the service registry.</span></span> <span data-ttu-id="76aae-115">Однако эти сведения со временем устаревают, и в результате клиенты больше не могут обнаружить экземпляры служб.</span><span class="sxs-lookup"><span data-stu-id="76aae-115">However, that information eventually goes out of date and clients can no longer discover service instances.</span></span> <span data-ttu-id="76aae-116">Поэтому реестр служб представляет собой кластер серверов, которые используют протокол репликации для обеспечения согласованности.</span><span class="sxs-lookup"><span data-stu-id="76aae-116">Consequently, a service registry consists of a cluster of servers that use a replication protocol to maintain consistency.</span></span>

<span data-ttu-id="76aae-117">В некоторых средах развертывания микрослужб (называемых кластерами и рассматриваемых далее) функция обнаружения служб встроена.</span><span class="sxs-lookup"><span data-stu-id="76aae-117">In some microservice deployment environments (called clusters, to be covered in a later section), service discovery is built-in.</span></span> <span data-ttu-id="76aae-118">Например, служба контейнеров Azure в среде Kubernetes (AKS) обрабатывает регистрацию и отмену регистрации экземпляров служб.</span><span class="sxs-lookup"><span data-stu-id="76aae-118">For example, an Azure Container Service with Kubernetes (AKS) environment can handle service instance registration and deregistration.</span></span> <span data-ttu-id="76aae-119">Она также запускает прокси-сервер в каждом узле кластера, который выступает в роли маршрутизатора обнаружения на стороне сервера.</span><span class="sxs-lookup"><span data-stu-id="76aae-119">It also runs a proxy on each cluster host that plays the role of server-side discovery router.</span></span>
