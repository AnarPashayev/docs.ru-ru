# <a name="appendix-a---transactions"></a><span data-ttu-id="5fd22-103">Приложение а. транзакции</span><span class="sxs-lookup"><span data-stu-id="5fd22-103">Appendix A - Transactions</span></span>

<span data-ttu-id="5fd22-104">Windows Communication Foundation (WCF) поддерживает распределенные транзакции, что позволяет выполнять атомарные операции в нескольких службах.</span><span class="sxs-lookup"><span data-stu-id="5fd22-104">Windows Communication Foundation (WCF) supports distributed transactions, allowing you to perform atomic operations across multiple services.</span></span> <span data-ttu-id="5fd22-105">Эта функция основана на [координатор распределенных транзакций Майкрософт](/previous-versions/windows/desktop/ms684146(v=vs.85)).</span><span class="sxs-lookup"><span data-stu-id="5fd22-105">This functionality is based on the [Microsoft Distributed Transaction Coordinator](/previous-versions/windows/desktop/ms684146(v=vs.85)).</span></span>

<span data-ttu-id="5fd22-106">В современных микрослужбах такой тип автоматизированной обработки распределенных транзакций невозможен.</span><span class="sxs-lookup"><span data-stu-id="5fd22-106">In the newer microservices landscape, this type of automated distributed transaction processing isn't possible.</span></span> <span data-ttu-id="5fd22-107">Существует слишком много различных технологий, включая реляционные базы данных, NoSQL хранилища данных и системы обмена сообщениями.</span><span class="sxs-lookup"><span data-stu-id="5fd22-107">There are too many different technologies involved, including relational databases, NoSQL data stores, and messaging systems.</span></span> <span data-ttu-id="5fd22-108">Кроме того, в одной среде могут использоваться сочетания операционных систем, языков программирования и платформ.</span><span class="sxs-lookup"><span data-stu-id="5fd22-108">There might also be a mix of operating systems, programming languages, and frameworks in use in a single environment.</span></span>

<span data-ttu-id="5fd22-109">Распределенная транзакция WCF является реализацией того, что называется [двухфазной фиксацией (2PC)](https://en.wikipedia.org/wiki/Two-phase_commit_protocol).</span><span class="sxs-lookup"><span data-stu-id="5fd22-109">WCF distributed transaction is an implementation of what is known as a [two-phase commit (2PC)](https://en.wikipedia.org/wiki/Two-phase_commit_protocol).</span></span> <span data-ttu-id="5fd22-110">2PC транзакции можно реализовать вручную путем координирования сообщений между службами, создания открытых транзакций в каждой службе и отправки сообщений фиксации или отката в зависимости от успешности или сбоя.</span><span class="sxs-lookup"><span data-stu-id="5fd22-110">You can implement 2PC transactions manually by coordinating messages across services, creating open transactions within each service, and sending commit or rollback messages, depending upon success or failure.</span></span> <span data-ttu-id="5fd22-111">Однако сложность, связанная с управлением 2PC, может значительно увеличить экспоненциальное развитие систем.</span><span class="sxs-lookup"><span data-stu-id="5fd22-111">However, the complexity involved in managing 2PC can increase exponentially as systems evolve.</span></span> <span data-ttu-id="5fd22-112">Открытые транзакции хранят блокировки баз данных, которые могут негативно повлиять на производительность, или, что еще хуже, привести к взаимоблокировкам между службами.</span><span class="sxs-lookup"><span data-stu-id="5fd22-112">Open transactions hold database locks that can negatively affect performance, or, worse, cause cross-service deadlocks.</span></span>

<span data-ttu-id="5fd22-113">По возможности лучше избегать распределенных транзакций.</span><span class="sxs-lookup"><span data-stu-id="5fd22-113">If possible, it's best to avoid distributed transactions altogether.</span></span> <span data-ttu-id="5fd22-114">Если два элемента данных связаны так, чтобы требовать атомарных обновлений, рассмотрите возможность их обработки с помощью одной и той же службы.</span><span class="sxs-lookup"><span data-stu-id="5fd22-114">If two items of data are so linked as to require atomic updates, consider handling them both with the same service.</span></span> <span data-ttu-id="5fd22-115">Примените эти атомарные изменения с помощью одного запроса или сообщения к этой службе.</span><span class="sxs-lookup"><span data-stu-id="5fd22-115">Apply those atomic changes by using a single request or message to that service.</span></span>

<span data-ttu-id="5fd22-116">Если это невозможно, в качестве альтернативы можно использовать [шаблон Saga](https://microservices.io/patterns/data/saga.html).</span><span class="sxs-lookup"><span data-stu-id="5fd22-116">If that isn't possible, then one alternative is to use the [Saga pattern](https://microservices.io/patterns/data/saga.html).</span></span> <span data-ttu-id="5fd22-117">В Saga обновления обрабатываются последовательно. После завершения каждого обновления запускается следующий.</span><span class="sxs-lookup"><span data-stu-id="5fd22-117">In a saga, updates are processed sequentially; as each update succeeds, the next one is triggered.</span></span> <span data-ttu-id="5fd22-118">Эти триггеры могут распространяться из службы в службу или управляются координатором Saga или Orchestrator.</span><span class="sxs-lookup"><span data-stu-id="5fd22-118">These triggers can be propagated from service to service, or managed by a saga coordinator or orchestrator.</span></span> <span data-ttu-id="5fd22-119">Если обновление завершается сбоем в любой момент, службы, которые уже выполнили их обновления, применяют определенную логику для их изменения.</span><span class="sxs-lookup"><span data-stu-id="5fd22-119">If an update fails at any point during the process, the services that have already completed their updates apply specific logic to reverse them.</span></span>

<span data-ttu-id="5fd22-120">Другой вариант — использовать проектирование на основе домена (DDD) и разделение ответственности за использование команд и запросов (CQRS), как описано в [электронной документации по микрослужбам .NET](../microservices/microservice-ddd-cqrs-patterns/index.md).</span><span class="sxs-lookup"><span data-stu-id="5fd22-120">Another option is to use Domain Driven Design (DDD) and Command/Query Responsibility Segregation (CQRS), as described in the [.NET Microservices e-book](../microservices/microservice-ddd-cqrs-patterns/index.md).</span></span> <span data-ttu-id="5fd22-121">В частности, использование событий предметной области или [источников событий](https://martinfowler.com/eaaDev/EventSourcing.html) может помочь обеспечить согласованность обновлений, если они не применяются немедленно.</span><span class="sxs-lookup"><span data-stu-id="5fd22-121">In particular, using domain events or [event sourcing](https://martinfowler.com/eaaDev/EventSourcing.html) can help to ensure that updates are consistently, if not immediately, applied.</span></span>
