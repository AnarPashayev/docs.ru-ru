# <a name="state-and-data-in-docker-applications"></a><span data-ttu-id="0de21-103">Состояние и данные в приложениях Docker</span><span class="sxs-lookup"><span data-stu-id="0de21-103">State and data in Docker applications</span></span>

<span data-ttu-id="0de21-104">Представьте себе контейнер как экземпляр процесса.</span><span class="sxs-lookup"><span data-stu-id="0de21-104">In most cases, you can think of a container as an instance of a process.</span></span> <span data-ttu-id="0de21-105">Процесс не находится в неизменном состоянии.</span><span class="sxs-lookup"><span data-stu-id="0de21-105">A process does not maintain persistent state.</span></span> <span data-ttu-id="0de21-106">И хотя контейнер может записывать данные в локальное хранилище, предполагать, что экземпляр всегда будет на месте — это все равно, что надеяться на постоянство одной ячейки памяти.</span><span class="sxs-lookup"><span data-stu-id="0de21-106">While a container can write to its local storage, assuming that an instance will be around indefinitely is like assuming that a single location in memory will be durable.</span></span> <span data-ttu-id="0de21-107">Следует считать, что образы контейнеров, как и процессы, имеют несколько экземпляров и в конечном итоге будут завершены. Если они управляются с помощью оркестратора контейнеров, они могут перемещаться между узлами или виртуальными машинами.</span><span class="sxs-lookup"><span data-stu-id="0de21-107">Container images, like processes, should be assumed to have multiple instances and that they will eventually be killed; if they're managed with a container orchestrator, it should be assumed that they might get moved from one node or VM to another.</span></span>

<span data-ttu-id="0de21-108">Для управления постоянными данными в приложениях Docker используются следующие решения:</span><span class="sxs-lookup"><span data-stu-id="0de21-108">The following solutions are used to manage persistent data in Docker applications:</span></span>

<span data-ttu-id="0de21-109">Из узла Docker в качестве [томов Docker](https://docs.docker.com/engine/admin/volumes/):</span><span class="sxs-lookup"><span data-stu-id="0de21-109">From the Docker host, as [Docker Volumes](https://docs.docker.com/engine/admin/volumes/):</span></span>

- <span data-ttu-id="0de21-110">**Тома** хранятся в той части файловой системы узла, которая находится под управлением Docker.</span><span class="sxs-lookup"><span data-stu-id="0de21-110">**Volumes** are stored in an area of the host filesystem that's managed by Docker.</span></span>

- <span data-ttu-id="0de21-111">**Подключения привязок** можно сопоставить с любой папкой в файловой системе узла, поэтому доступом невозможно управлять из процесса Docker, что может представлять угрозу для безопасности, поскольку контейнер может получить доступ к конфиденциальным папкам операционной системы.</span><span class="sxs-lookup"><span data-stu-id="0de21-111">**Bind mounts** can map to any folder in the host filesystem, so access can't be controlled from a Docker process and can pose a security risk as a container could access sensitive OS folders.</span></span>

- <span data-ttu-id="0de21-112">**Подключения tmpfs** аналогичны виртуальным папкам, которые существуют только в памяти узла и никогда не записываются в файловую систему.</span><span class="sxs-lookup"><span data-stu-id="0de21-112">**tmpfs mounts** are like virtual folders that only exist in the host's memory and are never written to the filesystem.</span></span>

<span data-ttu-id="0de21-113">Из удаленного хранилища:</span><span class="sxs-lookup"><span data-stu-id="0de21-113">From remote storage:</span></span>

- <span data-ttu-id="0de21-114">[Служба хранилища Azure](https://azure.microsoft.com/documentation/services/storage/), предоставляющая геораспределенное хранилище для долгосрочного хранения данных контейнеров.</span><span class="sxs-lookup"><span data-stu-id="0de21-114">[Azure Storage](https://azure.microsoft.com/documentation/services/storage/) provides geo-distributable storage, providing a good long-term persistence solution for containers.</span></span>

- <span data-ttu-id="0de21-115">Удаленные реляционные базы данных, например [База данных SQL Azure](https://azure.microsoft.com/services/sql-database/), базы данных NoSQL, такие как [Azure Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/introduction), или службы кэша, такие как [Redis](https://redis.io/).</span><span class="sxs-lookup"><span data-stu-id="0de21-115">Remote relational databases like [Azure SQL Database](https://azure.microsoft.com/services/sql-database/), NoSQL databases like [Azure Cosmos DB](https://docs.microsoft.com/azure/cosmos-db/introduction), or cache services like [Redis](https://redis.io/).</span></span>

<span data-ttu-id="0de21-116">Из контейнера Docker:</span><span class="sxs-lookup"><span data-stu-id="0de21-116">From the Docker container:</span></span>

- <span data-ttu-id="0de21-117">Docker имеет функцию под названием *оверлейная файловая система*.</span><span class="sxs-lookup"><span data-stu-id="0de21-117">Docker provides a feature named the *overlay file system*.</span></span> <span data-ttu-id="0de21-118">Эта функция выполняет копирование при записи, при котором обновленная информация сохраняется в корневой файловой системе контейнера.</span><span class="sxs-lookup"><span data-stu-id="0de21-118">This feature implements a copy-on-write task that stores updated information to the root file system of the container.</span></span> <span data-ttu-id="0de21-119">Эта информация "накладывается" на изначальный образ, на котором базируется контейнер.</span><span class="sxs-lookup"><span data-stu-id="0de21-119">That information "lays on top of" the original image on which the container is based.</span></span> <span data-ttu-id="0de21-120">Если удалить контейнер из системы, эти изменения будут утеряны.</span><span class="sxs-lookup"><span data-stu-id="0de21-120">If the container is deleted from the system, those changes are lost.</span></span> <span data-ttu-id="0de21-121">Поэтому, хоть и возможно сохранить состояние контейнера в его локальном хранилище, создание системы на базе этой функции будет противоречить конструкции контейнера, состояние которого по умолчанию не отслеживается.</span><span class="sxs-lookup"><span data-stu-id="0de21-121">Therefore, while it's possible to save the state of a container within its local storage, designing a system based on this feature would conflict with the premise of container design, which by default is stateless.</span></span>

- <span data-ttu-id="0de21-122">Тем не менее тома Docker являются предпочтительным способом обработки локальных данных в Docker.</span><span class="sxs-lookup"><span data-stu-id="0de21-122">However, Docker Volumes is now the preferred way to handle local data in Docker.</span></span> <span data-ttu-id="0de21-123">Если вам требуются дополнительные сведения о хранении в контейнерах, ознакомьтесь с разделами [Docker storage drivers](https://docs.docker.com/engine/userguide/storagedriver/) (Драйверы хранилища Docker) и [About images, containers, and storage drivers](https://docs.docker.com/engine/userguide/storagedriver/imagesandcontainers/) (Основные сведения об образах, контейнерах и драйверах хранилища).</span><span class="sxs-lookup"><span data-stu-id="0de21-123">If you need more information about storage in containers, check on [Docker storage drivers](https://docs.docker.com/engine/userguide/storagedriver/) and [About images, containers, and storage drivers](https://docs.docker.com/engine/userguide/storagedriver/imagesandcontainers/).</span></span>

<span data-ttu-id="0de21-124">Ниже подробно описаны эти варианты.</span><span class="sxs-lookup"><span data-stu-id="0de21-124">The following provides additional detail about these options.</span></span>

<span data-ttu-id="0de21-125">**Тома** — это каталоги из ОС узла, сопоставленные с каталогами в контейнерах.</span><span class="sxs-lookup"><span data-stu-id="0de21-125">**Volumes** are directories mapped from the host OS to directories in containers.</span></span> <span data-ttu-id="0de21-126">Если код в контейнере имеет доступ к каталогу, на самом деле он обращается к каталогу на ОС узла.</span><span class="sxs-lookup"><span data-stu-id="0de21-126">When code in the container has access to the directory, that access is actually to a directory on the host OS.</span></span> <span data-ttu-id="0de21-127">Этот каталог не привязан к времени существования самого контейнера, находится под управлением Docker и изолирован от основных функциональных возможностей хост-компьютера.</span><span class="sxs-lookup"><span data-stu-id="0de21-127">This directory is not tied to the lifetime of the container itself, and the directory is managed by Docker and isolated from the core functionality of the host machine.</span></span> <span data-ttu-id="0de21-128">Поэтому тома данных хранят данные независимо от контейнера.</span><span class="sxs-lookup"><span data-stu-id="0de21-128">Thus, data volumes are designed to persist data independently of the life of the container.</span></span> <span data-ttu-id="0de21-129">Если удалить контейнер или образ из узла Docker, данные из томов данных не будут удалены.</span><span class="sxs-lookup"><span data-stu-id="0de21-129">If you delete a container or an image from the Docker host, the data persisted in the data volume is not deleted.</span></span>

<span data-ttu-id="0de21-130">Тома могут быть именованными или анонимными (по умолчанию).</span><span class="sxs-lookup"><span data-stu-id="0de21-130">Volumes can be named or anonymous (the default).</span></span> <span data-ttu-id="0de21-131">Именованные тома — это следующий этап развития **контейнеров томов данных**. Они упрощают обмен данными между контейнерами.</span><span class="sxs-lookup"><span data-stu-id="0de21-131">Named volumes are the evolution of **Data Volume Containers** and make it easy to share data between containers.</span></span> <span data-ttu-id="0de21-132">Тома также поддерживают драйверы томов, позволяющие хранить данные на удаленных узлах и т. д.</span><span class="sxs-lookup"><span data-stu-id="0de21-132">Volumes also support volume drivers that allow you to store data on remote hosts, among other options.</span></span>

<span data-ttu-id="0de21-133">**Подключения привязки** доступны довольно давно и позволяют сопоставлять любую папку с точкой подключения в контейнере.</span><span class="sxs-lookup"><span data-stu-id="0de21-133">**Bind mounts** have been available for a long time and allow the mapping of any folder to a mount point in a container.</span></span> <span data-ttu-id="0de21-134">Подключения привязки имеют больше ограничений, чем тома, и с ними связаны некоторые проблемы безопасности, поэтому тома являются рекомендуемым способом.</span><span class="sxs-lookup"><span data-stu-id="0de21-134">Bind mounts have more limitations than volumes and some important security issues, so volumes are the recommended option.</span></span>

<span data-ttu-id="0de21-135">**Подключения `tmpfs`** являются виртуальными папками, которые существуют только в памяти узла и никогда не записываются в файловую систему.</span><span class="sxs-lookup"><span data-stu-id="0de21-135">**`tmpfs` mounts** are virtual folders that live only in the host's memory and are never written to the filesystem.</span></span> <span data-ttu-id="0de21-136">Они быстрые и безопасные, но используют ресурсы памяти и предназначены только для временных данных.</span><span class="sxs-lookup"><span data-stu-id="0de21-136">They are fast and secure but use memory and are only meant for non-persistent data.</span></span>

<span data-ttu-id="0de21-137">Как показано на рисунке 4-5, обычные тома Docker могут храниться за пределами самих контейнерами, но в физических границах сервера узла или виртуальной машины.</span><span class="sxs-lookup"><span data-stu-id="0de21-137">As shown in Figure 4-5, regular Docker volumes can be stored outside of the containers themselves but within the physical boundaries of the host server or VM.</span></span> <span data-ttu-id="0de21-138">Тем не менее контейнеры Docker с одного сервера узла или виртуальной машины не могут обращаться к тому на другом сервере узла или виртуальной машине.</span><span class="sxs-lookup"><span data-stu-id="0de21-138">However, Docker containers cannot access a volume from one host server or VM to another.</span></span> <span data-ttu-id="0de21-139">Другими словами, с помощью этих томов невозможно управлять данными контейнеров, которые выполняются на разных узлах Docker, хотя такой режим работы можно реализовать с помощью драйвера томов, который поддерживает удаленные узлы.</span><span class="sxs-lookup"><span data-stu-id="0de21-139">In other words, with these volumes, it isn't possible to manage data shared between containers that run on different Docker hosts, although it could be achieved with a volume driver that supports remote hosts.</span></span>

<span data-ttu-id="0de21-143">Кроме того, когда контейнеры Docker управляются оркестратором, контейнеры могут "перемещаться" между узлами в рамках оптимизации, выполняемой кластером.</span><span class="sxs-lookup"><span data-stu-id="0de21-143">In addition, when Docker containers are managed by an orchestrator, containers might "move" between hosts, depending on the optimizations performed by the cluster.</span></span> <span data-ttu-id="0de21-144">Поэтому тома данных не рекомендуется использовать для бизнес-данных.</span><span class="sxs-lookup"><span data-stu-id="0de21-144">Therefore, it isn't recommended that you use data volumes for business data.</span></span> <span data-ttu-id="0de21-145">Но это хороший инструмент для работы с файлами трассировки, временными файлами или подобными элементами, которые не влияют на целостность бизнес-данных.</span><span class="sxs-lookup"><span data-stu-id="0de21-145">But they are a good mechanism to work with trace files, temporal files, or similar, that will not impact business data consistency.</span></span>

<span data-ttu-id="0de21-146">**Удаленные источники данных и инструменты кэширования**, такие как база данных Azure SQL, Azure Cosmos DB или служба кэша, например Redis, можно использовать в упакованных в контейнеры приложениях так же, как они используются при разработке без контейнеров.</span><span class="sxs-lookup"><span data-stu-id="0de21-146">**Remote data sources and cache** tools like Azure SQL Database, Azure Cosmos DB, or a remote cache like Redis can be used in containerized applications the same way they are used when developing without containers.</span></span> <span data-ttu-id="0de21-147">Это проверенный способ хранения данных бизнес-приложений.</span><span class="sxs-lookup"><span data-stu-id="0de21-147">This is a proven way to store business application data.</span></span>

<span data-ttu-id="0de21-148">**Служба хранилища Azure**</span><span class="sxs-lookup"><span data-stu-id="0de21-148">**Azure Storage.**</span></span> <span data-ttu-id="0de21-149">Бизнес-данные, как правило, необходимо хранить во внешних ресурсах или базах данных, например в службе хранилища Azure.</span><span class="sxs-lookup"><span data-stu-id="0de21-149">Business data usually needs to be placed in external resources or databases, like Azure Storage.</span></span> <span data-ttu-id="0de21-150">Служба хранилища Azure предоставляет следующие службы в облаке:</span><span class="sxs-lookup"><span data-stu-id="0de21-150">Azure Storage provides the following services in the cloud:</span></span>
